CuboDW — Manual técnico

Motor OLAP (estilo Mondrian) escrito en Go: interpreta MDX, mapea el modelo multidimensional a un star schema relacional, genera SQL (5 dialectos), devuelve resultados tipados e incluye una aplicación visual de arrastrar y soltar con autenticación. Despliegue en microservicios Docker.

Cómo leer este manual. La sección 2 da el mapa de capas. Las secciones 5 y 6 documentan cada módulo (paquete Go). La 7 cubre la autenticación, la 8 la API REST, la 9 la cobertura MDX, la 10 la interfaz visual y la 11 los dialectos SQL. Los números de los ejemplos son reales, calculados sobre el cubo de demostración FoodMart (año 1997).

1. Visión general

CuboDW convierte preguntas analíticas (en MDX o en un modelo de consulta JSON tipado) en SQL ejecutado contra un data warehouse PostgreSQL, devolviendo una matriz de celdas tipadas (un CellSet). El cubo se describe con un esquema (Mondrian XML o YAML) que conecta conceptos lógicos (dimensiones, jerarquías, niveles, medidas) con tablas y columnas físicas. Sobre el motor están una UI visual y una capa de autenticación con roles.

2. Arquitectura

El código es un único módulo Go en tres capas: engine (el "Mondrian"), service (la lógica estilo Saiku) y web (HTTP/REST + UI). El flujo de una consulta MDX recorre las capas de arriba abajo:

   HTTP  →  web (middleware de autenticación)  →  handlers /saiku/api/*
       │
   MDX (texto)
       │  engine/mdx  ............... lexer + parser  →  AST
       ▼
   service/mdxeval  .............. resuelve ejes/miembros, reduce al modelo de query
       │  engine/query  ............ {medidas, niveles en filas/columnas, filtros}
       ▼
   engine/sql  ................... genera SQL (star schema, JOIN hecho↔dimensión, por Dialect)
       │  service/queryexec  ....... ejecuta en PostgreSQL (pgx) + caché de resultados
       ▼
   PostgreSQL  →  records  →  service/mdxeval (pivote)  →  CellSet
       ▲
   engine/metadata  ............. IR del cubo (alimentada por schema/mondrian|yaml)

servicediscover, queryexec, mdxevalDescubrimiento de metadatos, ejecución + caché, evaluación MDX → CellSet

3. Instalación y ejecución

Solo se necesita Docker + Docker Compose en el host. Go y Java no son necesarios: la compilación de Go corre en contenedor.

Levantar el stack

git clone https://github.com/davidkestering/projeto_golap.git && cd projeto_golap
make up            # docker compose: engine + postgres:16 (FoodMart cargado automáticamente)
curl localhost:8088/health        # {"status":"ok"}
curl localhost:8088/ready         # {"status":"ready"}  (conectado a Postgres)
# 👉 UI de arrastrar y soltar:  http://localhost:8088/ui/   (login demo: admin / admin)
make down          # derriba el stack

4. Conceptos OLAP (glosario)

TérminoSignificado
CuboConjunto de medidas analizables por dimensiones. Mapea a una tabla de hechos.
DimensiónEje de análisis (Tiempo, Producto, Tienda). Mapea a una tabla de dimensión unida a la de hechos por una clave foránea.
JerarquíaSecuencia de niveles dentro de una dimensión (p. ej. País → Estado → Ciudad). Una dimensión puede tener varias jerarquías (p. ej. Time por defecto y Weekly).
NivelUn grado de la jerarquía (p. ej. "Store State"). Mapea a una columna.
MiembroUn valor de un nivel (p. ej. "CA", "WA").
MedidaValor numérico agregable (p. ej. Unit Sales). Vive en la dimensión especial [Measures].
Miembro calculadoMedida/miembro definido por expresión en WITH MEMBER (p. ej. Profit = Store Sales − Store Cost).
Named setConjunto con nombre definido en WITH SET y reutilizado en los ejes.
MDXMultiDimensional eXpressions: lenguaje de consulta de cubos.
Star schemaModelo físico: una tabla de hechos central + tablas de dimensión unidas por FK=PK.
CellSetResultado de una consulta MDX: ejes (con posiciones) y celdas (valores).
Drill-throughBajar de una celda agregada a las filas de hechos crudas que la componen.
Nombre únicoIdentificador MDX entre corchetes, p. ej. [Time].[Year], [Measures].[Unit Sales].

5. Módulos del motor

5.1 engine metadata

internal/engine/metadata

Define la IR del modelo multidimensional, objetivo común de los loaders y fuente del descubrimiento y de la generación de SQL. Jerarquía de tipos: Schema → Cube → Dimension → Hierarchy → Level (+ Property) y Cube → Measure.

TipoCampos relevantes
CubeName, Caption, DefaultMeasure, Fact Relation, Dimensions, Measures
HierarchyName, HasAll, PrimaryKey, Table, Levels; EffectiveName(d) resuelve jerarquía por defecto vs nombrada
LevelName, Column, NameColumn, ParentColumn/NullParentValue (parent-child), Type, LevelType; DisplayColumn(), IsParentChild()
MeasureName, Column, Aggregator (sum|count|min|max|avg|distinct-count), FormatString, Expression

5.2 engine schema/mondrian

internal/engine/schema/mondrian

Carga esquemas Mondrian (XML v3) a la IR. Resuelve DimensionUsage contra dimensiones compartidas, dimensiones inline, múltiples jerarquías por dimensión, nameColumn/parentColumn (parent-child) y MeasureExpression (prefiere el dialecto postgres).

func LoadFile(path string) (*metadata.Schema, error)
func LoadBytes(b []byte)   (*metadata.Schema, error)

5.3 engine schema/yaml

internal/engine/schema/yaml

Carga cubos en el formato de autoría YAML a la misma IR. Cada dimensión YAML se aplana en una jerarquía única (con miembro "All"). Ver el formato completo en la sección 13.

5.4 engine query

internal/engine/query

Define la especificación de consulta tabular (pivote) y el formato del resultado tipado. El evaluador MDX (sección 6.3) produce esta misma estructura. Incluye el formateador de medidas query.Format (formatString).

type Query struct { Cube string; Measures []string; Rows, Columns []LevelRef; Filters []Filter; Totals bool }
type LevelRef struct { Dimension, Hierarchy, Level string }
type Column struct { Name, UniqueName, Kind, FormatString string }  // kind: level|measure

// Format aplica un formatString estilo Mondrian/VBA: "#,###.00", "#,###",
// "Standard", "Currency", "0.0%", … (agrupación de miles, decimales).
func Format(v float64, formatString string) string

5.5 engine sql + dialectos

internal/engine/sql

Genera SQL relacional a partir de una query.Query sobre un cubo (star schema). La generación la dirige un Dialect que abstrae las diferencias entre bases. Ver la lista de dialectos en la sección 11.

type Dialect interface {
    Name() string
    QuoteIdent(name string) string                 // "x" | `x` | [x]
    Placeholder(i int) string                      // $1 | ? | @p1 | :1
    CastText(expr) string;  CastFloat(expr) string
    InClause(col, members, startArg) (sql, args)   // ANY($n) o IN (…)
    SelectTop(n) string;    LimitSuffix(n) string  // TOP n / LIMIT n / FETCH FIRST
    AliasSep() string                              // " AS " | " " (Oracle)
}
func Build(d Dialect, cube *metadata.Cube, q query.Query) (*Statement, error)
func BuildLevelMembers(d Dialect, cube, ref, filters) (*Statement, error)
func BuildDrillthrough(d Dialect, cube, filters, maxrows) (*Statement, error)
func DialectByName(name string) (Dialect, error)

Ejemplo — Build (PostgreSQL)

SELECT "Time"."the_year", sum("f"."unit_sales")::float8
FROM "sales_fact_1997" AS "f"
JOIN "time_by_day" AS "Time" ON "f"."time_id" = "Time"."time_id"
GROUP BY "Time"."the_year"
ORDER BY 1

Los niveles con nameColumn (p. ej. parent-child) muestran el nombre en vez de la clave vía DisplayColumn(). Las dimensiones en snowflake (vía <Join>) devuelven un error claro.

5.6 engine mdx

internal/engine/mdx

Implementa el lexer, AST y parser de MDX, fiel a la gramática de Mondrian (MdxParser.jj). Solo hace parsing (texto → AST); la evaluación está en mdxeval.

func ParseQuery(input string)      (*Query, error)    // SELECT … FROM … WHERE …
func ParseExpression(input string) (Exp, error)

Decisión de diseño fiel a Mondrian: miembros como [Measures].[Unit Sales] son un único Id multi-segmento; solo se vuelven FunCall con paréntesis (método/función). Soporta WITH MEMBER/WITH SET, rangos (m1 : m2), CASE y CAST.

5.7 apoyo demo

internal/demo

Embebe el esquema FoodMart (Mondrian XML) usado como cubo por defecto cuando no se configura ningún esquema.

6. Módulos de servicio

6.1 service discover

internal/service/discover

Sirve metadatos del esquema cargado (rol del OlapMetaExplorer de Saiku). Modela el árbol conexión → catálogo → esquema → cubo sobre un único esquema en memoria.

6.2 service queryexec

internal/service/queryexec

Planifica (genera SQL) y ejecuta consultas de cubo contra PostgreSQL, con caché de resultados (FIFO por SQL+args), enumeración de miembros y drill-through.

func New(pool *pgxpool.Pool, cacheSize int) *Service
func (s *Service) Run(ctx, cube, st *sql.Statement)         (*query.Result, error)
func (s *Service) EnumerateLevel(ctx, cube, ref, filters)   ([]string, error)
func (s *Service) Drillthrough(ctx, cube, filters, maxrows) (*query.Result, error)
func (s *Service) CacheStats() (hits, misses, size int)
func (s *Service) CacheClear()

Cada celda de medida se formatea con el FormatString de la columna (p. ej. 565,238.13).

6.3 service mdxeval

internal/service/mdxeval

El evaluador MDX. Reduce el AST al modelo de query (niveles + medidas + filtros), ejecuta vía queryexec y pivota los records en un CellSet. Resuelve las posiciones de cada eje de forma independiente (en el contexto del slicer), habilitando orden/límite/filtro por medida, miembros calculados, named sets, rangos, inteligencia de tiempo (time.go), múltiples jerarquías y parent-child.

func Evaluate(ctx, cube *metadata.Cube, q *mdx.Query, exec *queryexec.Service) (*CellSet, error)

6.4 apoyo auth

internal/auth

Store de usuarios (contraseñas en bcrypt, persistencia opcional en JSON) y tokens de sesión firmados (HMAC-SHA256, stateless). Roles admin/user; siembra admin/admin cuando está vacío. Uso en la sección 7.

func NewStore(file string) (*Store, error)
func (s *Store) Register(username, password string, role Role) error
func (s *Store) Authenticate(username, password string) (User, bool)
func SignToken(secret []byte, u User, ttl time.Duration) string
func VerifyToken(secret []byte, token string) (User, bool)

7. Autenticación y roles

La autenticación está activa por defecto (CUBODW_AUTH_ENABLED=true). Un middleware protege todos los endpoints bajo /saiku/api/*, excepto los públicos. La sesión es una cookie firmada (HMAC) — sin estado en el servidor; las contraseñas usan bcrypt.

RecursoDetalle
Rolesadmin (todo) y user (consulta/descubrimiento)
Admin sembradoadmin / admin (cámbialo en producción)
Registroabierto: cualquiera crea un usuario user y queda autenticado
Rutas públicas/health, /ready, /saiku/api/info, /saiku/api/auth/*, /ui/*
Solo adminAcciones administrativas (p. ej. POST /saiku/api/cache/clear) → 403 para user
Cookiecubodw_session, HttpOnly, SameSite=Lax, TTL 12h

Flujo (cURL con cookies)

# sin login, un endpoint protegido → 401
curl -i localhost:8088/saiku/api/discover                       # HTTP 401

# login (guarda la cookie de sesión)
curl -c cookies.txt -X POST localhost:8088/saiku/api/auth/login \
  -H 'Content-Type: application/json' -d '{"username":"admin","password":"admin"}'
# {"username":"admin","role":"admin"}

# con la cookie, la API responde
curl -b cookies.txt localhost:8088/saiku/api/discover           # HTTP 200

# registro abierto (crea rol user, ya autenticado)
curl -X POST localhost:8088/saiku/api/auth/register \
  -H 'Content-Type: application/json' -d '{"username":"maria","password":"s3creto"}'
# {"username":"maria","role":"user"}
Producción. Define CUBODW_AUTH_SECRET (si no, se usa un secreto de desarrollo, con aviso en el log), persiste usuarios con CUBODW_USERS_FILE, cambia la contraseña del admin y sirve detrás de HTTPS (para habilitar el atributo Secure en la cookie).

8. API REST

Todas las rutas están bajo /saiku/api/* (contrato compatible con Saiku). Base local: http://localhost:8088. Con la auth activa, envía la cookie de sesión (sección 7) — los ejemplos omiten -b cookies.txt por brevedad.

GET /health · /ready · /saiku/api/info

Salud (liveness), prontitud (hace ping a Postgres) y metadatos del servicio (incluye cache). Públicos.

POST /saiku/api/auth/register · /login · /logout · GET /me

Registro abierto (crea user), login (cookie), logout (limpia la cookie) e identidad actual.

GET /saiku/api/discover · …/{cube}/metadata

Árbol conexión → catálogo → esquema → cubos y, por cubo, dimensiones (con jerarquías y niveles) y medidas.

POST /saiku/api/query · /saiku/api/query/preview

Ejecuta (o solo valida + genera SQL, en /preview) el modelo de query JSON (sin MDX). Con "totals":true devuelve también la fila de totales.

$ curl -X POST localhost:8088/saiku/api/query -H 'Content-Type: application/json' -d '{
  "cube":"Sales", "rows":[{"dimension":"Time","level":"Year"}],
  "measures":["Unit Sales","Store Sales"]
}'
{"rows":[[{"value":1997,"formatted":"1997"},
          {"value":266773,"formatted":"266,773"},
          {"value":565238.13,"formatted":"565,238.13"}]]}

POST /saiku/api/query/drillthrough

Devuelve las filas de hechos crudas detrás de una celda agregada (filtros = miembros de la celda; maxrows limita).

POST /saiku/api/mdx/parse · /saiku/api/mdx/execute

parse devuelve el AST (ejes, slicer, fórmulas, forma canónica). execute hace parse, evalúa y devuelve el CellSet (con grid para 2 ejes).

GET /saiku/api/ai/cubes · /ai/schema/{cube} · POST /ai/query

AI Query API: consulta cubos sin MDX, con esquema auto-descriptivo (miembros de muestra) y validación tipada. Ante un error de nombre, devuelve {status, field, available:[…]} para autocorrección.

GET /saiku/api/schemas · POST /schemas/validate · /schemas (admin) · DELETE /schemas/{name} (admin)

Gestor de cubos: lista, valida (dry-run), añade y elimina esquemas en tiempo de ejecución. Cuerpo: {"content":"…","format":"yaml|xml"} (YAML de autoría o Mondrian XML). Los nombres de cubo/esquema se normalizan (MAYÚSCULAS; acentos transliterados — á→a, ç→c; espacios/símbolos eliminados) y los repetidos reciben sufijo incremental V1/V2/… — nunca hay colisión.

GET /saiku/api/cache · POST /saiku/api/cache/clear (admin)

Métricas del caché (hits, misses, size) y limpieza (rol admin).

Códigos: 400 nombre inválido · 401 no autenticado · 403 requiere admin · 404 cubo inexistente · 503 sin base.

9. Cobertura MDX

El motor cubre un subconjunto amplio de MDX. Las tablas listan lo soportado, con ejemplos y resultado esperado sobre FoodMart (1997).

9.1 Estructura de la query

[WITH <fórmulas: MEMBER | SET>]
SELECT <conjunto> [NON EMPTY] ON COLUMNS [, <conjunto> ON ROWS …]
FROM [Cubo]
[WHERE <slicer>]

9.7 Formato de medidas

El formatString de cada medida se aplica al resultado: Unit Sales (Standard) → 266,773; Store Sales (#,###.00) → 565,238.13; Sales Count (#,###) → 86,837.

9.8 Operadores y precedencia

De menor a mayor: OR / XOR / : → AND → NOT → comparaciones → + - || → * / → unario + - → . (miembro/método). Soporta CASE … WHEN … THEN … ELSE … END y CAST.

11. Dialectos SQL

La generación de SQL la dirige un Dialect (sección 5.5). Cinco dialectos están implementados y cubiertos por golden tests. El runtime ejecuta vía pgx (PostgreSQL); apuntar a los demás requiere conectar el driver respectivo y cargar los datos en esa base (paso de despliegue) — la capa de generación de SQL ya está lista.

DialectoQuotingPlaceholderTexto / FloatLímite
PostgreSQL"x"$1(e)::text / (e)::float8LIMIT n
MySQL/MariaDB`x`?CAST(e AS CHAR/DOUBLE)LIMIT n
DuckDB"x"?CAST(e AS VARCHAR/DOUBLE)LIMIT n
SQL Server[x]@p1CAST(e AS NVARCHAR/FLOAT)SELECT TOP n
Oracle"x":1CAST(e AS VARCHAR2/BINARY_DOUBLE)FETCH FIRST n ROWS ONLY

Resuelve por nombre con sql.DialectByName("mysql"|"duckdb"|"sqlserver"|"oracle"|"postgres"). Oracle, por ejemplo, omite el AS antes del alias de tabla (FROM "t" "f").

12. Ejemplos de extremo a extremo

12.1 Desglose por dimensión

POST /saiku/api/query
{"cube":"Sales","rows":[{"dimension":"Customers","level":"Country"}],
 "measures":["Unit Sales","Store Sales","Sales Count","Customer Count"]}
→ USA | 266,773 | 565,238.13 | 86,837 | 5,581

12.2 Grilla 2-D vía MDX (CrossJoin)

{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS,
         CrossJoin([Gender].[Gender].Members,[Marital Status].[Marital Status].Members) ON ROWS
         FROM [Sales]"}
→ F/M:65336  F/S:66222  M/M:66460  M/S:68755   (suma 266773)

12.3 Porcentaje del total (miembro calculado + Sum)

{"mdx":"WITH MEMBER [Measures].[Pct] AS
           [Measures].[Unit Sales] / Sum([Store].[Store State].Members,[Measures].[Unit Sales]) * 100
         SELECT {[Measures].[Unit Sales],[Measures].[Pct]} ON COLUMNS,
                [Store].[Store State].Members ON ROWS FROM [Sales]"}
→ CA:74748|28.02   OR:67659|25.36   WA:124366|46.62   (Σ ≈ 100%)

12.4 Jerarquía nombrada (Weekly)

{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS,
         [Time].[Weekly].[Week].Members ON ROWS
         FROM [Sales] WHERE [Time].[1997]"}
→ 52 semanas (1..52)   — vs [Time].[Quarter].Members → 4 trimestres

12.5 Mostrar todos los miembros vs NON EMPTY

… [Store].[Store State].Members ON ROWS …            → 10 filas (vacías incluidas)
… NON EMPTY [Store].[Store State].Members ON ROWS …  → 3 filas (CA, OR, WA)

13. Formatos de esquema

13.1 Mondrian XML (v3)

<Schema name="FoodMart">
  <Dimension name="Store">
    <Hierarchy hasAll="true" primaryKey="store_id">
      <Table name="store"/>
      <Level name="Store Country" column="store_country" uniqueMembers="true"/>
      <Level name="Store State"   column="store_state"/>
    </Hierarchy>
  </Dimension>
  <Cube name="Sales" defaultMeasure="Unit Sales">
    <Table name="sales_fact_1997"/>
    <DimensionUsage name="Store" source="Store" foreignKey="store_id"/>

  </Cube>
</Schema>

13.2 YAML de autoría

schema: FoodMart
cubes:
  - name: Sales
    fact: sales_fact_1997
    defaultMeasure: Unit Sales
    measures:
      - {name: Unit Sales, column: unit_sales, agg: sum}
      - {name: Store Sales, column: store_sales, agg: sum, format: "#,###.00"}

        foreignKey: time_id

Carga un esquema fijo en el arranque vía CUBODW_SCHEMA=/ruta/archivo.xml (o .yml), o añade cubos en tiempo de ejecución desde la UI (botón ⬢ Cubos, admin) / API POST /saiku/api/schemas — ver secciones 8 y 10. La persistencia ya viene activada en docker-compose.yml (volúmenes locales en deploy/data/): los cubos añadidos y los usuarios registrados sobreviven a reinicios.

14. Limitaciones (aún no soportado)

El motor es funcional y cubre la mayoría de las consultas de BI, pero no alcanza la paridad total con Mondrian. Las construcciones no soportadas devuelven un error claro:

  • Mostrar todos los miembros en un CrossJoin (multi-binding aún enumera vía hechos).
  • Dimensiones en snowflake (definidas por <Join>, p. ej. Product → product_class).
  • Parent-child solo como lista plana (por nombre): navegación de árbol y rollup recursivo aún no.
  • Ejecución en runtime fuera de PostgreSQL: los dialectos MySQL/DuckDB/SQL Server/Oracle existen en la generación de SQL, pero el runtime usa pgx (conectar otro driver + datos es un paso de despliegue).
