CuboDW — Manual técnico

Motor OLAP (estilo Mondrian) escrito en Go: interpreta MDX, mapea el modelo multidimensional a un star schema relacional, genera SQL (PostgreSQL) y devuelve resultados tipados. Despliegue en microservicios Docker.

Cómo leer este manual. La sección 2 da el mapa de las capas. Las secciones 5 y 6 documentan cada módulo (paquete Go) con sus tipos, funciones y ejemplos. La 7 describe la API REST y la 8 la cobertura de MDX. Los números de los ejemplos son reales, calculados sobre el cubo de demostración FoodMart (año 1997).

1. Visión general

CuboDW convierte preguntas analíticas (en MDX o en un modelo de consulta JSON tipado) en SQL ejecutado en un data warehouse PostgreSQL, devolviendo una matriz de celdas tipadas (un CellSet). El cubo se describe mediante un esquema (Mondrian XML o YAML) que vincula conceptos lógicos (dimensiones, jerarquías, niveles, medidas) con tablas y columnas físicas.

2. Arquitectura

El código es un único módulo Go en tres capas: engine (el "Mondrian"), service (la lógica estilo Saiku) y web (HTTP/REST). Una consulta MDX fluye de arriba hacia abajo:

   MDX (texto)
       │  engine/mdx  ............... lexer + parser  →  AST
       ▼
   service/mdxeval  .............. resuelve ejes/miembros, reduce al modelo de consulta
       │  engine/query  ............ {medidas, niveles en filas/columnas, filtros}
       ▼
   engine/sql  ................... genera SQL (star schema, JOIN hecho↔dimensión)
       │  service/queryexec  ....... lo ejecuta en PostgreSQL (pgx)
       ▼
   PostgreSQL  →  records  →  service/mdxeval (pivot)  →  CellSet
       ▲
   engine/metadata  ............. IR del cubo (alimentada por schema/mondrian|yaml)
CapaPaquetesResponsabilidad
enginemetadata, schema/mondrian, schema/yaml, query, sql, mdxModelo multidimensional, parsing MDX, generación de SQL
servicediscover, queryexec, mdxevalDescubrimiento de metadatos, ejecución, evaluación MDX → CellSet
webwebServidor HTTP, endpoints /saiku/api/*
apoyoconfig, version, demoConfiguración por env, versión, esquema FoodMart embebido

3. Instalación y ejecución

Solo se necesita Docker + Docker Compose en el host. Go y Java no son necesarios: la compilación de Go se ejecuta en un contenedor.

Levantar el stack

make up            # docker compose: engine + postgres:16 (FoodMart auto-cargado)
curl localhost:8088/health        # {"status":"ok"}
curl localhost:8088/ready         # {"status":"ready"}  (conectado a Postgres)
curl localhost:8088/saiku/api/info
make down          # baja el stack

CLI (cmd/cubodw)

ComandoDescripción
cubodw serve-engineInicia el servidor HTTP del motor (flag --addr)
cubodw healthcheck --url …GET al endpoint de salud; sale 0 si 200 (usado por el HEALTHCHECK de Docker)
cubodw versionImprime la versión

Configuración (variables de entorno — paquete config)

VariableDefaultDescripción
CUBODW_HTTP_ADDR:8080Dirección de escucha HTTP (en el contenedor)
CUBODW_PG_DSN (o PG_DSN)—DSN de PostgreSQL. Vacío: /ready reporta no-db
CUBODW_SCHEMA—Ruta del esquema (.xml=Mondrian, .yml/.yaml=YAML). Vacío: FoodMart embebido

El host expone el engine en localhost:8088 (mapeado al 8080 del contenedor). PostgreSQL está en localhost:5432 (usuario/contraseña cubodw, base de datos foodmart).

4. Conceptos OLAP (glosario)

TérminoSignificado
CuboConjunto de medidas analizables por dimensiones. Mapea a una tabla de hechos.
DimensiónEje de análisis (Tiempo, Producto, Tienda). Mapea a una tabla de dimensión enlazada a los hechos por una clave foránea.
JerarquíaSecuencia de niveles dentro de una dimensión (p. ej. País → Estado → Ciudad).
NivelUn grado de la jerarquía (p. ej. "Store State"). Mapea a una columna.
MiembroUn valor de un nivel (p. ej. "CA", "WA").
MedidaValor numérico agregable (p. ej. Unit Sales = suma de unidades). Vive en la dimensión especial [Measures].
MDXMultiDimensional eXpressions: lenguaje de consulta de cubos.
Star schemaModelo físico: una tabla de hechos central + tablas de dimensión enlazadas por FK=PK.
CellSetResultado de una consulta MDX: ejes (con posiciones) y celdas (valores).
Nombre únicoIdentificador MDX entre corchetes, p. ej. [Time].[Year], [Measures].[Unit Sales].

5. Módulos del motor

5.1 engine metadata

internal/engine/metadata

Define la IR (representación interna) del modelo multidimensional, destino común de los loaders y fuente del descubrimiento y de la generación de SQL. Jerarquía de tipos: Schema → Cube → Dimension → Hierarchy → Level (+ Property) y, en paralelo, Cube → Measure.

Tipos principales

TipoCampos relevantes
SchemaName, Cubes []*Cube, SharedDimensions []*Dimension
CubeName, Caption, DefaultMeasure, Fact Relation, Dimensions, Measures
DimensionName, Type (StandardDimension|TimeDimension), ForeignKey, Hierarchies
HierarchyName, HasAll, AllMemberName, PrimaryKey, Table Relation, Levels
LevelName, Column, Type, LevelType, UniqueMembers, Properties
MeasureName, Column, Aggregator (sum|count|min|max|avg|distinct-count), FormatString, Expression
RelationSchema, Table, Alias — referencia a una tabla física

Funciones

5.2 engine schema/mondrian

internal/engine/schema/mondrian

Carga esquemas Mondrian (XML v3) a la IR. Resuelve DimensionUsage contra dimensiones compartidas, dimensiones inline y MeasureExpression (prefiere el dialecto postgres).

func LoadFile(path string) (*metadata.Schema, error)
func LoadBytes(b []byte)   (*metadata.Schema, error)
func Load(r io.Reader)     (*metadata.Schema, error)
schema, err := mondrian.LoadFile("FoodMart.xml")
cube, _ := schema.FindCube("Sales")
// cube.Fact.Table == "sales_fact_1997"
// cube.Measures: Unit Sales, Store Cost, Store Sales, Sales Count, Customer Count, …
// cube.Dimensions: Store, Time, Product, Customers, Gender, …

5.4 engine query

internal/engine/query

Define la especificación de una consulta tabular (pivot) y el formato del resultado tipado. Es el precursor del "thin query": describe medidas, niveles en filas/columnas y filtros sin exponer MDX. El evaluador MDX (sección 6.3) produce esta misma estructura.

type Query struct {
    Cube     string       `json:"cube"`
    Measures []string     `json:"measures"`
    Rows     []LevelRef   `json:"rows,omitempty"`
    Columns  []LevelRef   `json:"columns,omitempty"`
    Filters  []Filter     `json:"filters,omitempty"`
}
type LevelRef struct { Dimension, Hierarchy, Level string }
type Filter   struct { Dimension, Hierarchy, Level string; Members []string }

type Result struct {
    Cube    string     `json:"cube"`
    SQL     string     `json:"sql"`      // SQL generada (transparencia)
    Columns []Column   `json:"columns"`  // {name, uniqueName, kind: level|measure}
    Rows    [][]Cell   `json:"rows"`     // {value, formatted}
}

5.5 engine sql

internal/engine/sql

Genera SQL relacional a partir de una query.Query sobre un cubo (star schema). Dialecto actual: PostgreSQL.

type Dialect interface { QuoteIdent(name string) string; Name() string }
type Postgres struct{}                       // dialecto PostgreSQL
type Statement struct { SQL string; Args []any; Columns []query.Column }

func Build(d Dialect, cube *metadata.Cube, q query.Query) (*Statement, error)
func BuildLevelMembers(d Dialect, cube *metadata.Cube, ref query.LevelRef, filters []query.Filter) (*Statement, error)

Ejemplo — Build

Consulta: medida Unit Sales por Time.Year →

SELECT "Time"."the_year", sum("f"."unit_sales")::float8
FROM "sales_fact_1997" AS "f"
JOIN "time_by_day" AS "Time" ON "f"."time_id" = "Time"."time_id"
GROUP BY "Time"."the_year"
ORDER BY 1

Los filtros se convierten en predicados (columna)::text = ANY($n); las medidas numéricas salen como float8 y los conteos como enteros. Las dimensiones en snowflake (vía <Join>) devuelven un error claro.

Ejemplo — BuildLevelMembers

Enumera todos los miembros de un nivel desde la tabla de dimensión (base de la semántica "mostrar todos los miembros"):

SELECT DISTINCT "Store"."store_state"
FROM "store" AS "Store"
WHERE ("Store"."store_country")::text = ANY($1)   -- p. ej. ['USA']
ORDER BY 1

5.6 engine mdx

internal/engine/mdx

Implementa el lexer, AST y parser del lenguaje MDX, fiel a la gramática de Mondrian (MdxParser.jj). Solo hace el parsing (texto → AST); la evaluación está en mdxeval.

func Tokenize(input string)        ([]Token, error)   // lexer
func ParseQuery(input string)      (*Query, error)    // SELECT … FROM … WHERE …
func ParseExpression(input string) (Exp, error)       // expresión aislada

AST

Todo es una Exp. Las llamadas a función/operadores se convierten en FunCall{Name, Syntax, Args}; los identificadores son Id con segmentos. Decisión de diseño fiel a Mondrian: miembros como [Measures].[Unit Sales] y [Store].[USA].Children son un único Id multi-segmento; solo se convierten en FunCall cuando hay paréntesis (método/función).

TipoDescripción
QueryFormulas []*Formula (WITH), Axes []*Axis, Cube string, Slicer Exp (WHERE)
AxisNonEmpty bool, Exp, Ordinal int (0=COLUMNS,1=ROWS,…)
FormulaIsMember bool, Name *Id, Exp
FunCallName, Syntax (Function/Method/Property/Infix/Prefix/Postfix/Braces/Parentheses/Case/Cast), Args []Exp
IdSegments []Segment (Name, Quoted, Key)
LiteralesNumericLiteral, StringLiteral, NullLiteral

Cada nodo tiene String() que reconstruye una forma canónica (útil para depurar y para generar MDX).

q, _ := mdx.ParseQuery(`SELECT {[Measures].[Unit Sales]} ON COLUMNS,
                        [Store].[Store Country].Members ON ROWS
                        FROM [Sales] WHERE [Time].[1997]`)
// q.Cube == "Sales";  len(q.Axes) == 2
// q.Axes[1].Exp.String() == "[Store].[Store Country].Members"
// q.Slicer.String()      == "[Time].[1997]"

6.1 service discover

6.2 service queryexec

internal/service/queryexec

Planifica (genera SQL) y ejecuta consultas de cubo contra PostgreSQL, devolviendo un query.Result tipado.

func New(pool *pgxpool.Pool) *Service
func (s *Service) Plan(cube, q query.Query)              (*sql.Statement, error)  // solo genera SQL (validación)
func (s *Service) Run(ctx, cube, st *sql.Statement)      (*query.Result, error)   // ejecuta
func (s *Service) EnumerateLevel(ctx, cube, ref, filters) ([]string, error)       // miembros vía dimensión
func (s *Service) HasDB() bool                                                     // ¿hay conexión?

Las medidas numéricas (sumas) regresan como float8 → número JSON; los conteos como enteros. EnumerateLevel lista los miembros de un nivel desde la tabla de dimensión (no de los hechos).

6.3 service mdxeval

internal/service/mdxeval

El evaluador MDX. Reduce el AST al modelo de consulta (niveles + medidas + filtros), ejecuta vía queryexec y pivotea los records en un CellSet. Estrategia: resuelve las posiciones de cada eje de forma independiente (en el contexto del slicer), lo que hace que el ordenamiento/límite/filtrado por medida sea fiel a Mondrian y habilita los miembros calculados.

func Evaluate(ctx, cube *metadata.Cube, q *mdx.Query, exec *queryexec.Service) (*CellSet, error)

Resultado — CellSet

type CellSet struct {
    Cube  string     `json:"cube"`
    SQL   string     `json:"sql"`
    Axes  []Axis     `json:"axes"`    // {ordinal, name, positions:[{members:[{caption,uniqueName}]}]}
    Cells []Cell     `json:"cells"`   // {coords:[…], value, formatted}
    Grid  *Grid      `json:"grid"`    // conveniencia para 2 ejes: columnHeaders, rowHeaders, rows[][]
}

7. API REST

Todas las rutas viven bajo /saiku/api/* (contrato compatible con Saiku, para conectar una UI en el futuro). Base local: http://localhost:8088.

GET /health · /ready · /saiku/api/info

Salud (liveness), preparación (hace ping a Postgres) y metadatos del servicio.

$ curl localhost:8088/saiku/api/info
{"name":"cubodw-engine","version":"0.0.1","engine":"mondrian-go (en construcción)",
 "connection":"foodmart","schema":"FoodMart","cubes":7}

GET /saiku/api/discover

El árbol conexión → catálogo → esquema → cubos.

$ curl localhost:8088/saiku/api/discover
[{"name":"foodmart","catalogs":[{"name":"FoodMart","schemas":[{"name":"FoodMart",
  "cubes":[{"name":"Sales","uniqueName":"[Sales]","caption":"Sales", … }, … ]}]}]}]

Dimensiones (con jerarquías y niveles) y medidas de un cubo. (También existe …/{cube}/dimensions para solo la lista de dimensiones.)

8. Cobertura MDX

El motor cubre un subconjunto amplio de MDX. Las tablas siguientes listan lo soportado, con ejemplos y el resultado esperado sobre FoodMart (1997).

8.1 Estructura de la consulta

[WITH <fórmulas>]
SELECT <conjunto> [NON EMPTY] ON COLUMNS [, <conjunto> ON ROWS …]
FROM [Cubo]
[WHERE <slicer>]
ConstrucciónEjemplo
Ejes… ON COLUMNS, … ON ROWS, … ON AXIS(2), … ON 0
NON EMPTYNON EMPTY [Store].[Store State].Members ON ROWS (poda miembros sin hechos)
Slicer (WHERE)WHERE ([Time].[1997], [Gender].[F])

8.4 Miembros calculados (WITH MEMBER)

Aritmética sobre medidas y funciones escalares, evaluadas en Go por celda.

RecursoEjemploResultado (USA, 1997)
AritméticaWITH MEMBER [Measures].[Profit] AS [Measures].[Store Sales] - [Measures].[Store Cost]339610.90
IIf(cond, x, y)IIf([Measures].[Unit Sales] > 80000, 1, 0)1 / 0
CoalesceEmpty(a, b)CoalesceEmpty([Measures].[X],[Measures].[Store Sales])el primero definido
Sum / Avg / Count / Aggregate (sobre un conjunto)[Measures].[Unit Sales] / Sum([Store].[Store State].Members,[Measures].[Unit Sales]) * 100% del total (CA 28.02, OR 25.36, WA 46.62)
Subtotal en el contexto correcto. En Sum(set, medida), el subtotal se precalcula agrupado por las demás dimensiones del grid (no un escalar global). P. ej., en un grid Género × Estado, el % por estado suma 100% dentro de cada género.

8.5 Operadores y precedencia

De menor a mayor precedencia: OR / XOR / : → AND → NOT → comparaciones (= <> < > <= >=, IS) → + - || → * / → unario + - → . (miembro/método). Soporta CASE … WHEN … THEN … ELSE … END y CAST.

9. Ejemplos de extremo a extremo

9.1 Desglose por dimensión (records)

POST /saiku/api/query
{"cube":"Sales","rows":[{"dimension":"Customers","level":"Country"}],
 "measures":["Unit Sales","Store Sales","Sales Count","Customer Count"]}

→ USA | 266773 | 565238.13 | 86837 | 5581

9.2 Grid 2-D vía MDX

POST /saiku/api/mdx/execute
{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS,
         CrossJoin([Gender].[Gender].Members,[Marital Status].[Marital Status].Members) ON ROWS
         FROM [Sales]"}

→ F / M : 65336      F / S : 66222
  M / M : 66460      M / S : 68755     (suma = 266773)

9.3 Top-N + ordenamiento componibles

{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS,
         Head(Order([Store].[Store State].Members,[Measures].[Unit Sales],BDESC),2) ON ROWS
         FROM [Sales]"}

→ WA : 124366      CA : 74748

9.4 Porcentaje del total (miembro calculado + Sum de conjunto)

{"mdx":"WITH MEMBER [Measures].[Pct] AS
           [Measures].[Unit Sales] / Sum([Store].[Store State].Members,[Measures].[Unit Sales]) * 100
         SELECT {[Measures].[Unit Sales],[Measures].[Pct]} ON COLUMNS,
                [Store].[Store State].Members ON ROWS FROM [Sales]"}

→ CA : 74748 | 28.02      OR : 67659 | 25.36      WA : 124366 | 46.62   (Σ ≈ 100%)

9.5 Mostrar todos los miembros vs NON EMPTY

… [Store].[Store State].Members ON ROWS …            → 10 filas (BC, DF, Jalisco… vacías)
… NON EMPTY [Store].[Store State].Members ON ROWS …  → 3 filas (CA, OR, WA)

10. Formatos de esquema

10.1 Mondrian XML (v3)

<Schema name="FoodMart">
  <Dimension name="Store">
    <Hierarchy hasAll="true" primaryKey="store_id">
      <Table name="store"/>
      <Level name="Store Country" column="store_country" uniqueMembers="true"/>
      <Level name="Store State"   column="store_state"/>
    </Hierarchy>
  </Dimension>
  <Cube name="Sales" defaultMeasure="Unit Sales">
    <Table name="sales_fact_1997"/>
    <DimensionUsage name="Store" source="Store" foreignKey="store_id"/>
    <Measure name="Unit Sales" column="unit_sales" aggregator="sum"/>
  </Cube>
</Schema>

10.2 YAML de autoría

schema: FoodMart
cubes:
  - name: Sales
    fact: sales_fact_1997
    defaultMeasure: Unit Sales
    measures:
      - {name: Unit Sales, column: unit_sales, agg: sum}
      - {name: Store Sales, column: store_sales, agg: sum, format: "#,###.00"}
    dimensions:
      - name: Time
        foreignKey: time_id
        table: time_by_day
        primaryKey: time_id
        type: time
        levels:
          - {name: Year, column: the_year, type: Numeric, levelType: TimeYears}
          - {name: Quarter, column: quarter, levelType: TimeQuarters}

Cargue cualquiera vía CUBODW_SCHEMA=/ruta/archivo.xml (o .yml).

11. Limitaciones (aún no soportado)

El motor es funcional y cubre la mayoría de las consultas de BI, pero no tiene paridad total con Mondrian. Las construcciones no soportadas devuelven un error claro:

  • Min/Max sobre conjuntos en miembros calculados.
  • Mostrar todos los miembros en CrossJoin (multi-binding aún enumera vía hechos).
  • Named sets (WITH SET), rangos (m1 : m2), funciones de tiempo (YTD, ParallelPeriod, Lag/Lead).
  • Dimensiones en snowflake (definidas por <Join>, p. ej. Product → product_class).
  • Múltiples jerarquías por dimensión, miembros padre-hijo.
  • Dialectos distintos de PostgreSQL.
