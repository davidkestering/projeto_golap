CuboDW — Technical manual

An OLAP engine (Mondrian-style) written in Go: it interprets MDX, maps the multidimensional model onto a relational star schema, generates SQL (5 dialects), returns typed results, and ships a drag-and-drop visual app with authentication. Deployed as Docker microservices.

How to read this manual. Section 2 maps the layers. Sections 5 and 6 document each module (Go package). Section 7 covers authentication, 8 the REST API, 9 MDX coverage, 10 the visual UI and 11 the SQL dialects. Example numbers are real, computed over the FoodMart demo cube (year 1997).

1. Overview

CuboDW turns analytical questions (in MDX or a typed JSON query model) into SQL run against a PostgreSQL warehouse, returning a matrix of typed cells (a CellSet). The cube is described by a schema (Mondrian XML or YAML) that maps logical concepts (dimensions, hierarchies, levels, measures) to physical tables and columns. On top of the engine sit a visual UI and an authentication-with-roles layer.

2. Architecture

The code is a single Go module in three layers: engine (the "Mondrian"), service (Saiku-style logic) and web (HTTP/REST + UI). An MDX query flows top to bottom:

   HTTP  →  web (authentication middleware)  →  handlers /saiku/api/*
       │
   MDX (text)
       │  engine/mdx  ............... lexer + parser  →  AST
       ▼
   service/mdxeval  .............. resolves axes/members, reduces to the query model
       │  engine/query  ............ {measures, levels on rows/columns, filters}
       ▼
   engine/sql  ................... generates SQL (star schema, fact↔dimension JOIN, per Dialect)
       │  service/queryexec  ....... runs on PostgreSQL (pgx) + result cache
       ▼
   PostgreSQL  →  records  →  service/mdxeval (pivot)  →  CellSet
       ▲
   engine/metadata  ............. cube IR (fed by schema/mondrian|yaml)

enginemetadata, schema/mondrian, schema/yaml, query, sql (+ dialect), mdxMultidimensional model, MDX parsing, multi-dialect SQL generation
servicediscover, queryexec, mdxevalMetadata discovery, execution + cache, MDX → CellSet evaluation
webweb (+ embedded web/ui)HTTP server, authentication (middleware), /saiku/api/* endpoints, /ui/ SPA
supportconfig, version, demo, authEnv config, version, embedded FoodMart, user store + tokens

3. Install & run

Only Docker + Docker Compose are required on the host. Go and Java are not: the Go build runs in a container.

Bring up the stack

git clone https://github.com/davidkestering/projeto_golap.git && cd projeto_golap
make up            # docker compose: engine + postgres:16 (FoodMart auto-loaded)
curl localhost:8088/health        # {"status":"ok"}
curl localhost:8088/ready         # {"status":"ready"}  (connected to Postgres)
# 👉 drag-and-drop UI:  http://localhost:8088/ui/   (demo login: admin / admin)
make down          # tear the stack down

4. OLAP concepts (glossary)

TermMeaning
CubeA set of measures analysable by dimensions. Maps to a fact table.
DimensionAn axis of analysis (Time, Product, Store). Maps to a dimension table joined to the fact by a foreign key.
HierarchyA sequence of levels within a dimension (e.g. Country → State → City). A dimension may have several hierarchies (e.g. Time default and Weekly).
LevelOne rung of the hierarchy (e.g. "Store State"). Maps to a column.
MemberA value of a level (e.g. "CA", "WA").
MeasureAn aggregatable numeric value (e.g. Unit Sales). Lives in the special [Measures] dimension.
Calculated memberA measure/member defined by an expression in WITH MEMBER (e.g. Profit = Store Sales − Store Cost).
Named setA named set defined in WITH SET and reused on axes.
MDXMultiDimensional eXpressions: the cube query language.
Star schemaPhysical model: one central fact table + dimension tables joined by FK=PK.
CellSetThe result of an MDX query: axes (with positions) and cells (values).
Drill-throughGoing from an aggregated cell down to the raw fact rows behind it.
Unique nameBracketed MDX identifier, e.g. [Time].[Year], [Measures].[Unit Sales].

5. Engine modules

5.1 engine metadata

internal/engine/metadata

Defines the IR of the multidimensional model, the common target of the loaders and the source for discovery and SQL generation. Type hierarchy: Schema → Cube → Dimension → Hierarchy → Level (+ Property), plus Cube → Measure.

TypeRelevant fields
CubeName, Caption, DefaultMeasure, Fact Relation, Dimensions, Measures
HierarchyName, HasAll, PrimaryKey, Table, Levels; EffectiveName(d) resolves default vs named hierarchy
LevelName, Column, NameColumn, ParentColumn/NullParentValue (parent-child), Type, LevelType; DisplayColumn(), IsParentChild()
MeasureName, Column, Aggregator (sum|count|min|max|avg|distinct-count), FormatString, Expression

5.2 engine schema/mondrian

internal/engine/schema/mondrian

Loads Mondrian schemas (XML v3) into the IR. Resolves DimensionUsage against shared dimensions, inline dimensions, multiple hierarchies per dimension, nameColumn/parentColumn (parent-child) and MeasureExpression (prefers the postgres dialect).

func LoadFile(path string) (*metadata.Schema, error)
func LoadBytes(b []byte)   (*metadata.Schema, error)

5.3 engine schema/yaml

internal/engine/schema/yaml

Loads cubes in the YAML authoring format into the same IR. Each YAML dimension is flattened into a single hierarchy (with an "All" member). See the full format in section 13.

5.4 engine query

internal/engine/query

Defines the tabular (pivot) query spec and the typed result format. The MDX evaluator (section 6.3) produces this same structure. Includes the measure formatter query.Format (formatString).

type Query struct { Cube string; Measures []string; Rows, Columns []LevelRef; Filters []Filter; Totals bool }
type LevelRef struct { Dimension, Hierarchy, Level string }
type Column struct { Name, UniqueName, Kind, FormatString string }  // kind: level|measure

// Format applies a Mondrian/VBA-style formatString: "#,###.00", "#,###",
// "Standard", "Currency", "0.0%", … (thousands grouping, decimal places).
func Format(v float64, formatString string) string

5.5 engine sql + dialects

internal/engine/sql

Generates relational SQL from a query.Query over a cube (star schema). Generation is driven by a Dialect that abstracts database differences. See the dialect list in section 11.

type Dialect interface {
    Name() string
    QuoteIdent(name string) string                 // "x" | `x` | [x]
    Placeholder(i int) string                      // $1 | ? | @p1 | :1
    CastText(expr) string;  CastFloat(expr) string
    InClause(col, members, startArg) (sql, args)   // ANY($n) or IN (…)
    SelectTop(n) string;    LimitSuffix(n) string  // TOP n / LIMIT n / FETCH FIRST
    AliasSep() string                              // " AS " | " " (Oracle)
}
func Build(d Dialect, cube *metadata.Cube, q query.Query) (*Statement, error)
func BuildLevelMembers(d Dialect, cube, ref, filters) (*Statement, error)
func BuildDrillthrough(d Dialect, cube, filters, maxrows) (*Statement, error)
func DialectByName(name string) (Dialect, error)

Example — Build (PostgreSQL)

SELECT "Time"."the_year", sum("f"."unit_sales")::float8
FROM "sales_fact_1997" AS "f"
JOIN "time_by_day" AS "Time" ON "f"."time_id" = "Time"."time_id"
GROUP BY "Time"."the_year"
ORDER BY 1

Levels with a nameColumn (e.g. parent-child) display the name instead of the key via DisplayColumn(). Snowflake dimensions (via <Join>) return a clear error.

5.6 engine mdx

internal/engine/mdx

Implements the MDX lexer, AST and parser, faithful to Mondrian's grammar (MdxParser.jj). Parsing only (text → AST); evaluation lives in mdxeval.

func ParseQuery(input string)      (*Query, error)    // SELECT … FROM … WHERE …
func ParseExpression(input string) (Exp, error)

Mondrian-faithful design choice: members like [Measures].[Unit Sales] are a single multi-segment Id; they become a FunCall only with parentheses (method/function). Supports WITH MEMBER/WITH SET, ranges (m1 : m2), CASE and CAST.

5.7 support demo

internal/demo

Embeds the FoodMart schema (Mondrian XML) used as the default cube when no schema is configured.

6. Service modules

6.1 service discover

internal/service/discover

Serves metadata for the loaded schema (Saiku's OlapMetaExplorer role). Models the connection → catalog → schema → cube tree over a single in-memory schema.

6.2 service queryexec

internal/service/queryexec

Plans (generates SQL) and executes cube queries against PostgreSQL, with a result cache (FIFO by SQL+args), member enumeration and drill-through.

func New(pool *pgxpool.Pool, cacheSize int) *Service
func (s *Service) Run(ctx, cube, st *sql.Statement)         (*query.Result, error)
func (s *Service) EnumerateLevel(ctx, cube, ref, filters)   ([]string, error)
func (s *Service) Drillthrough(ctx, cube, filters, maxrows) (*query.Result, error)
func (s *Service) CacheStats() (hits, misses, size int)
func (s *Service) CacheClear()

Each measure cell is formatted using the column's FormatString (e.g. 565,238.13).

6.3 service mdxeval

internal/service/mdxeval

The MDX evaluator. Reduces the AST to the query model (levels + measures + filters), runs it via queryexec and pivots the records into a CellSet. It resolves each axis's positions independently (in the slicer context), enabling sort/limit/filter by measure, calculated members, named sets, ranges, time intelligence (time.go), multiple hierarchies and parent-child.

func Evaluate(ctx, cube *metadata.Cube, q *mdx.Query, exec *queryexec.Service) (*CellSet, error)

6.4 support auth

internal/auth

A user store (bcrypt passwords, optional JSON persistence) and signed session tokens (HMAC-SHA256, stateless). Roles admin/user; seeds admin/admin when empty. Usage in section 7.

func NewStore(file string) (*Store, error)
func (s *Store) Register(username, password string, role Role) error
func (s *Store) Authenticate(username, password string) (User, bool)
func SignToken(secret []byte, u User, ttl time.Duration) string
func VerifyToken(secret []byte, token string) (User, bool)

7. Authentication & roles

Authentication is on by default (CUBODW_AUTH_ENABLED=true). A middleware protects every endpoint under /saiku/api/* except the public ones. The session is a signed cookie (HMAC) — no server state; passwords use bcrypt.

FeatureDetail
Rolesadmin (everything) and user (query/discovery)
Seeded adminadmin / admin (change in production)
Registrationopen: anyone creates a user and is logged in immediately
Public routes/health, /ready, /saiku/api/info, /saiku/api/auth/*, /ui/*
Admin-onlyAdmin actions (e.g. POST /saiku/api/cache/clear) → 403 for user
Cookiecubodw_session, HttpOnly, SameSite=Lax, 12h TTL

Flow (cURL with cookies)

# without login, a protected endpoint → 401
curl -i localhost:8088/saiku/api/discover                       # HTTP 401

# login (stores the session cookie)
curl -c cookies.txt -X POST localhost:8088/saiku/api/auth/login \
  -H 'Content-Type: application/json' -d '{"username":"admin","password":"admin"}'
# {"username":"admin","role":"admin"}

# with the cookie, the API responds
curl -b cookies.txt localhost:8088/saiku/api/discover           # HTTP 200

# open registration (creates a user role, already logged in)
curl -X POST localhost:8088/saiku/api/auth/register \
  -H 'Content-Type: application/json' -d '{"username":"mary","password":"s3cret"}'
# {"username":"mary","role":"user"}
Production. Set CUBODW_AUTH_SECRET (otherwise a dev secret is used, with a log warning), persist users with CUBODW_USERS_FILE, change the admin password and serve behind HTTPS (to enable the cookie's Secure attribute).

8. REST API

All routes live under /saiku/api/* (Saiku-compatible contract). Local base: http://localhost:8088. With auth on, send the session cookie (section 7) — examples omit -b cookies.txt for brevity.

GET /health · /ready · /saiku/api/info

Liveness, readiness (pings Postgres) and service metadata (includes cache). Public.

POST /saiku/api/auth/register · /login · /logout · GET /me

Open registration (creates user), login (cookie), logout (clears cookie) and current identity.

GET /saiku/api/discover · …/{cube}/metadata

The connection → catalog → schema → cubes tree and, per cube, dimensions (with hierarchies and levels) and measures.

POST /saiku/api/query · /saiku/api/query/preview

Runs (or just validates + generates SQL, in /preview) the JSON query model (no MDX). With "totals":true it also returns the totals row.

$ curl -X POST localhost:8088/saiku/api/query -H 'Content-Type: application/json' -d '{
  "cube":"Sales", "rows":[{"dimension":"Time","level":"Year"}],
  "measures":["Unit Sales","Store Sales"]
}'
{"rows":[[{"value":1997,"formatted":"1997"},
          {"value":266773,"formatted":"266,773"},
          {"value":565238.13,"formatted":"565,238.13"}]]}

POST /saiku/api/query/drillthrough

Returns the raw fact rows behind an aggregated cell (filters = cell members; maxrows caps).

POST /saiku/api/mdx/parse · /saiku/api/mdx/execute

parse returns the AST (axes, slicer, formulas, canonical form). execute parses, evaluates and returns the CellSet (with a grid for 2 axes).

GET /saiku/api/ai/cubes · /ai/schema/{cube} · POST /ai/query

AI Query API: queries cubes without MDX, with a self-describing schema (sample members) and typed validation. On a name error it returns {status, field, available:[…]} for self-correction.

GET /saiku/api/schemas · POST /schemas/validate · /schemas (admin) · DELETE /schemas/{name} (admin)

Cube manager: list, validate (dry-run), add and remove schemas at runtime. Body: {"content":"…","format":"yaml|xml"} (authoring YAML or Mondrian XML). Cube/schema names are normalized (UPPERCASE; accents transliterated — á→a, ç→c; spaces/symbols removed) and duplicates get an incremental V1/V2/… suffix — never a collision.

GET /saiku/api/cache · POST /saiku/api/cache/clear (admin)

Cache metrics (hits, misses, size) and clearing (admin role).

Codes: 400 invalid name · 401 unauthenticated · 403 admin required · 404 unknown cube · 503 no database.

The engine covers a broad MDX subset. Tables list what is supported, with examples and expected results over FoodMart (1997).

Subtotal in the right context. In Sum(set, measure) the subtotal is precomputed grouped by the other grid dimensions (not a global scalar). E.g. in a Gender × State grid, the per-state % sums to 100% within each gender.

9.7 Measure formatting

Each measure's formatString is applied to the result: Unit Sales (Standard) → 266,773; Store Sales (#,###.00) → 565,238.13; Sales Count (#,###) → 86,837.

9.8 Operators and precedence

From lowest to highest: OR / XOR / : → AND → NOT → comparisons → + - || → * / → unary + - → . (member/method). Supports CASE … WHEN … THEN … ELSE … END and CAST.

11. SQL dialects

SQL generation is driven by a Dialect (section 5.5). Five dialects are implemented and covered by golden tests. The runtime executes via pgx (PostgreSQL); targeting the others requires wiring the respective driver and loading data into that database (a deployment step) — the SQL generation layer is ready.

DialectQuotingPlaceholderText / FloatRow limit
PostgreSQL"x"$1(e)::text / (e)::float8LIMIT n
MySQL/MariaDB`x`?CAST(e AS CHAR/DOUBLE)LIMIT n
DuckDB"x"?CAST(e AS VARCHAR/DOUBLE)LIMIT n
SQL Server[x]@p1CAST(e AS NVARCHAR/FLOAT)SELECT TOP n
Oracle"x":1CAST(e AS VARCHAR2/BINARY_DOUBLE)FETCH FIRST n ROWS ONLY

Resolve by name with sql.DialectByName("mysql"|"duckdb"|"sqlserver"|"oracle"|"postgres"). Oracle, for instance, omits AS before a table alias (FROM "t" "f").

12. End-to-end examples

12.1 Breakdown by dimension

POST /saiku/api/query
{"cube":"Sales","rows":[{"dimension":"Customers","level":"Country"}],
 "measures":["Unit Sales","Store Sales","Sales Count","Customer Count"]}
→ USA | 266,773 | 565,238.13 | 86,837 | 5,581

12.2 2-D grid via MDX (CrossJoin)

{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS,
         CrossJoin([Gender].[Gender].Members,[Marital Status].[Marital Status].Members) ON ROWS
         FROM [Sales]"}
→ F/M:65336  F/S:66222  M/M:66460  M/S:68755   (sum 266773)

12.3 Percent of total (calculated member + Sum)

{"mdx":"WITH MEMBER [Measures].[Pct] AS
           [Measures].[Unit Sales] / Sum([Store].[Store State].Members,[Measures].[Unit Sales]) * 100
         SELECT {[Measures].[Unit Sales],[Measures].[Pct]} ON COLUMNS,
                [Store].[Store State].Members ON ROWS FROM [Sales]"}
→ CA:74748|28.02   OR:67659|25.36   WA:124366|46.62   (Σ ≈ 100%)

12.4 Named hierarchy (Weekly)

{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS,
         [Time].[Weekly].[Week].Members ON ROWS
         FROM [Sales] WHERE [Time].[1997]"}
→ 52 weeks (1..52)   — vs [Time].[Quarter].Members → 4 quarters

12.5 Show all members vs NON EMPTY

… [Store].[Store State].Members ON ROWS …            → 10 rows (empties included)
… NON EMPTY [Store].[Store State].Members ON ROWS …  → 3 rows (CA, OR, WA)

13. Schema formats

13.1 Mondrian XML (v3)

<Schema name="FoodMart">
  <Dimension name="Store">
    <Hierarchy hasAll="true" primaryKey="store_id">
      <Table name="store"/>
      <Level name="Store Country" column="store_country" uniqueMembers="true"/>
      <Level name="Store State"   column="store_state"/>
    </Hierarchy>
  </Dimension>
  <Cube name="Sales" defaultMeasure="Unit Sales">
    <Table name="sales_fact_1997"/>
    <DimensionUsage name="Store" source="Store" foreignKey="store_id"/>
    <Measure name="Unit Sales" column="unit_sales" aggregator="sum" formatString="Standard"/>
  </Cube>
</Schema>

13.2 Authoring YAML

schema: FoodMart
cubes:
  - name: Sales
    fact: sales_fact_1997
    defaultMeasure: Unit Sales
    measures:
      - {name: Unit Sales, column: unit_sales, agg: sum}
      - {name: Store Sales, column: store_sales, agg: sum, format: "#,###.00"}
    dimensions:
      - name: Time
        foreignKey: time_id
        table: time_by_day
        primaryKey: time_id
        type: time
        levels:
          - {name: Year, column: the_year, type: Numeric, levelType: TimeYears}
          - {name: Quarter, column: quarter, levelType: TimeQuarters}

Load a fixed schema at boot via CUBODW_SCHEMA=/path/file.xml (or .yml), or add cubes at runtime from the UI (⬢ Cubos button, admin) / API POST /saiku/api/schemas — see sections 8 and 10. Persistence is on by default in docker-compose.yml (local volumes under deploy/data/): added cubes and registered users survive restarts.

14. Limitations (not yet supported)

The engine is functional and covers most BI queries, but does not reach full Mondrian parity. Unsupported constructs return a clear error:

  • Showing all members in a CrossJoin (multi-binding still enumerates via the fact).
  • Snowflake dimensions (defined by <Join>, e.g. Product → product_class).
  • Parent-child only as a flat list (by name): tree navigation and recursive rollup are not done yet.
  • Runtime execution outside PostgreSQL: the MySQL/DuckDB/SQL Server/Oracle dialects exist at the SQL generation layer, but the runtime uses pgx (wiring another driver + data is a deployment step).
