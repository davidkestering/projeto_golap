CuboDW — Technical manual

An OLAP engine (Mondrian-style) written in Go: it interprets MDX, maps the multidimensional model onto a relational star schema, generates SQL (PostgreSQL) and returns typed results. Deployed as Docker microservices.

How to read this manual. Section 2 gives the map of layers. Sections 5 and 6 document each module (Go package) with its types, functions and examples. Section 7 describes the REST API and 8 the MDX coverage. Example numbers are real, computed over the demo FoodMart cube (year 1997).

1. Overview

CuboDW turns analytical questions (in MDX or a typed JSON query model) into SQL run against a PostgreSQL warehouse, returning a matrix of typed cells (a CellSet). The cube is described by a schema (Mondrian XML or YAML) that binds logical concepts (dimensions, hierarchies, levels, measures) to physical tables and columns.

ItemValue
LanguageGo 1.26
Target databasePostgreSQL 16 (reference dialect)
DeploymentDocker / docker compose (engine + postgres)

2. Architecture

The code is a single Go module in three layers: engine (the "Mondrian"), service (the Saiku-style logic) and web (HTTP/REST). An MDX query flows top to bottom:

   MDX (text)
       │  engine/mdx  ............... lexer + parser  →  AST
       ▼
   service/mdxeval  .............. resolves axes/members, reduces to the query model
       │  engine/query  ............ {measures, levels on rows/columns, filters}
       ▼
   engine/sql  ................... generates SQL (star schema, fact↔dimension JOIN)
       │  service/queryexec  ....... runs it on PostgreSQL (pgx)
       ▼
   PostgreSQL  →  records  →  service/mdxeval (pivot)  →  CellSet
       ▲
   engine/metadata  ............. cube IR (fed by schema/mondrian|yaml)
LayerPackagesResponsibility
enginemetadata, schema/mondrian, schema/yaml, query, sql, mdxMultidimensional model, MDX parsing, SQL generation
servicediscover, queryexec, mdxevalMetadata discovery, execution, MDX → CellSet evaluation
webwebHTTP server, /saiku/api/* endpoints
supportconfig, version, demoEnv config, version, embedded FoodMart schema

3. Install & run

Only Docker + Docker Compose are required on the host. Go and Java are not needed: the Go build runs in a container.

Bring up the stack

make up            # docker compose: engine + postgres:16 (FoodMart auto-loaded)
curl localhost:8088/health        # {"status":"ok"}
curl localhost:8088/ready         # {"status":"ready"}  (connected to Postgres)
curl localhost:8088/saiku/api/info
make down          # tear down

CLI (cmd/cubodw)

CommandDescription
cubodw serve-engineStarts the engine HTTP server (--addr flag)
cubodw healthcheck --url …GET on the health endpoint; exits 0 on 200 (used by the Docker HEALTHCHECK)
cubodw versionPrints the version

Configuration (environment variables — config package)

VariableDefaultDescription
CUBODW_HTTP_ADDR:8080HTTP listen address (inside the container)
CUBODW_PG_DSN (or PG_DSN)—PostgreSQL DSN. Empty: /ready reports no-db
CUBODW_SCHEMA—Schema path (.xml=Mondrian, .yml/.yaml=YAML). Empty: embedded FoodMart
CUBODW_CONNECTIONfoodmartLogical connection name exposed in discovery
The host exposes the engine at localhost:8088 (mapped to the container's 8080). PostgreSQL is at localhost:5432 (user/password cubodw, database foodmart).

4. OLAP concepts (glossary)

TermMeaning
CubeA set of measures analysable by dimensions. Maps to a fact table.
DimensionAn axis of analysis (Time, Product, Store). Maps to a dimension table linked to the fact by a foreign key.
HierarchyA sequence of levels within a dimension (e.g. Country → State → City).
LevelOne rung of the hierarchy (e.g. "Store State"). Maps to a column.
MemberA value of a level (e.g. "CA", "WA").
MeasureAn aggregatable numeric value (e.g. Unit Sales = sum of units). Lives in the special [Measures] dimension.
MDXMultiDimensional eXpressions: the cube query language.
Star schemaPhysical model: a central fact table + dimension tables linked by FK=PK.
CellSetThe result of an MDX query: axes (with positions) and cells (values).
Unique nameA bracketed MDX identifier, e.g. [Time].[Year], [Measures].[Unit Sales].

5.2 engine schema/mondrian

internal/engine/schema/mondrian

Loads Mondrian schemas (XML v3) into the IR. Resolves DimensionUsage against shared dimensions, inline dimensions, and MeasureExpression (prefers the postgres dialect).

func LoadFile(path string) (*metadata.Schema, error)
func LoadBytes(b []byte)   (*metadata.Schema, error)
func Load(r io.Reader)     (*metadata.Schema, error)
schema, err := mondrian.LoadFile("FoodMart.xml")
cube, _ := schema.FindCube("Sales")
// cube.Fact.Table == "sales_fact_1997"
// cube.Measures: Unit Sales, Store Cost, Store Sales, Sales Count, Customer Count, …
// cube.Dimensions: Store, Time, Product, Customers, Gender, …

5.3 engine schema/yaml

internal/engine/schema/yaml

Loads cubes in the concise YAML authoring format into the same IR. Each YAML dimension is flattened into a single hierarchy (with an "All" member).

5.4 engine query

internal/engine/query

Defines the tabular (pivot) query specification and the typed result format. It is the precursor of the "thin query": it describes measures, levels on rows/columns and filters without exposing MDX. The MDX evaluator (section 6.3) produces this very structure.

type Query struct {
    Cube     string       `json:"cube"`
    Measures []string     `json:"measures"`
    Rows     []LevelRef   `json:"rows,omitempty"`
    Columns  []LevelRef   `json:"columns,omitempty"`
    Filters  []Filter     `json:"filters,omitempty"`
}
type LevelRef struct { Dimension, Hierarchy, Level string }
type Filter   struct { Dimension, Hierarchy, Level string; Members []string }

type Result struct {
    Cube    string     `json:"cube"`
    SQL     string     `json:"sql"`      // generated SQL (transparency)
    Columns []Column   `json:"columns"`  // {name, uniqueName, kind: level|measure}
    Rows    [][]Cell   `json:"rows"`     // {value, formatted}
}

5.5 engine sql

internal/engine/sql

Generates relational SQL from a query.Query over a cube (star schema). Current dialect: PostgreSQL.

type Dialect interface { QuoteIdent(name string) string; Name() string }
type Postgres struct{}                       // PostgreSQL dialect
type Statement struct { SQL string; Args []any; Columns []query.Column }

func Build(d Dialect, cube *metadata.Cube, q query.Query) (*Statement, error)
func BuildLevelMembers(d Dialect, cube *metadata.Cube, ref query.LevelRef, filters []query.Filter) (*Statement, error)

Example — Build

Query: measure Unit Sales by Time.Year →

SELECT "Time"."the_year", sum("f"."unit_sales")::float8
FROM "sales_fact_1997" AS "f"
JOIN "time_by_day" AS "Time" ON "f"."time_id" = "Time"."time_id"
GROUP BY "Time"."the_year"
ORDER BY 1

Filters become (column)::text = ANY($n) predicates; numeric measures come out as float8 and counts as integers. Snowflake dimensions (via <Join>) return a clear error.

Example — BuildLevelMembers

Enumerates all members of a level from the dimension table (the basis of the "show all members" semantics):

SELECT DISTINCT "Store"."store_state"
FROM "store" AS "Store"
WHERE ("Store"."store_country")::text = ANY($1)   -- e.g. ['USA']
ORDER BY 1

5.6 engine mdx

internal/engine/mdx

Implements the MDX lexer, AST and parser, faithful to Mondrian's grammar (MdxParser.jj). It only parses (text → AST); evaluation lives in mdxeval.

func Tokenize(input string)        ([]Token, error)   // lexer
func ParseQuery(input string)      (*Query, error)    // SELECT … FROM … WHERE …
func ParseExpression(input string) (Exp, error)       // a standalone expression

AST

Everything is an Exp. Function/operator calls become FunCall{Name, Syntax, Args}; identifiers are Id with segments. A Mondrian-faithful design choice: members such as [Measures].[Unit Sales] and [Store].[USA].Children are a single multi-segment Id; they only become a FunCall when parentheses are present (method/function).

TypeDescription
QueryFormulas []*Formula (WITH), Axes []*Axis, Cube string, Slicer Exp (WHERE)
AxisNonEmpty bool, Exp, Ordinal int (0=COLUMNS,1=ROWS,…)
FormulaIsMember bool, Name *Id, Exp
FunCallName, Syntax (Function/Method/Property/Infix/Prefix/Postfix/Braces/Parentheses/Case/Cast), Args []Exp
IdSegments []Segment (Name, Quoted, Key)
LiteralsNumericLiteral, StringLiteral, NullLiteral

Every node has String() which rebuilds a canonical form (useful for debugging and for generating MDX).

q, _ := mdx.ParseQuery(`SELECT {[Measures].[Unit Sales]} ON COLUMNS,
                        [Store].[Store Country].Members ON ROWS
                        FROM [Sales] WHERE [Time].[1997]`)
// q.Cube == "Sales";  len(q.Axes) == 2
// q.Axes[1].Exp.String() == "[Store].[Store Country].Members"
// q.Slicer.String()      == "[Time].[1997]"

6. Service modules

6.1 service discover

internal/service/discover

Serves metadata for the loaded schema (the role of Saiku's OlapMetaExplorer). Models the connection → catalog → schema → cube tree over a single in-memory schema.

func New(connection string, schema *metadata.Schema) *Service
func (s *Service) Connection() string
func (s *Service) Catalog() string

func (s *Service) Cubes() []*metadata.Cube
func (s *Service) Cube(name string) (*metadata.Cube, bool)

6.2 service queryexec

internal/service/queryexec

Plans (generates SQL) and executes cube queries against PostgreSQL, returning a typed query.Result.

func New(pool *pgxpool.Pool) *Service
func (s *Service) Plan(cube, q query.Query)              (*sql.Statement, error)  // generate SQL only (validation)
func (s *Service) Run(ctx, cube, st *sql.Statement)      (*query.Result, error)   // execute
func (s *Service) EnumerateLevel(ctx, cube, ref, filters) ([]string, error)       // members via dimension
func (s *Service) HasDB() bool                                                     // is a DB configured?

Numeric measures (sums) come back as float8 → JSON number; counts as integers. EnumerateLevel lists the members of a level from the dimension table (not the fact).

6.3 service mdxeval

internal/service/mdxeval

The MDX evaluator. It reduces the AST to the query model (levels + measures + filters), executes via queryexec and pivots the records into a CellSet. Strategy: it resolves each axis's positions independently (in the slicer context), which makes ordering/limiting/filtering by measure faithful to Mondrian and enables calculated members.

func Evaluate(ctx, cube *metadata.Cube, q *mdx.Query, exec *queryexec.Service) (*CellSet, error)

Result — CellSet

type CellSet struct {
    Cube  string     `json:"cube"`
    SQL   string     `json:"sql"`
    Axes  []Axis     `json:"axes"`    // {ordinal, name, positions:[{members:[{caption,uniqueName}]}]}
    Cells []Cell     `json:"cells"`   // {coords:[…], value, formatted}
    Grid  *Grid      `json:"grid"`    // convenience for 2 axes: columnHeaders, rowHeaders, rows[][]
}

7. REST API

All routes live under /saiku/api/* (a Saiku-compatible contract, to plug a UI in later). Local base: http://localhost:8088.

GET /health · /ready · /saiku/api/info

Liveness, readiness (pings Postgres) and service metadata.

$ curl localhost:8088/saiku/api/info
{"name":"cubodw-engine","version":"0.0.1","engine":"mondrian-go (under construction)",
 "connection":"foodmart","schema":"FoodMart","cubes":7}

GET /saiku/api/discover

The connection → catalog → schema → cubes tree.

$ curl localhost:8088/saiku/api/discover
[{"name":"foodmart","catalogs":[{"name":"FoodMart","schemas":[{"name":"FoodMart",
  "cubes":[{"name":"Sales","uniqueName":"[Sales]","caption":"Sales", … }, … ]}]}]}]

GET /saiku/api/discover/{conn}/{catalog}/{schema}/{cube}/metadata

Dimensions (with hierarchies and levels) and measures of a cube. (There is also …/{cube}/dimensions for just the dimension list.)

$ curl localhost:8088/saiku/api/discover/foodmart/FoodMart/FoodMart/Sales/metadata
{"name":"Sales","uniqueName":"[Sales]","defaultMeasure":"Unit Sales",
 "dimensions":[{"name":"Time","hierarchies":[{"levels":[
     {"name":"Year","uniqueName":"[Time].[Year]"}, … ]}]}, … ],
 "measures":[{"name":"Unit Sales","uniqueName":"[Measures].[Unit Sales]", … }, … ]}

POST /saiku/api/query · /saiku/api/query/preview

Executes (or, on /preview, only validates + generates SQL) the JSON query model (no MDX).

$ curl -X POST localhost:8088/saiku/api/query -H 'Content-Type: application/json' -d '{
  "cube":"Sales",
  "rows":[{"dimension":"Time","level":"Year"}],
  "measures":["Unit Sales","Store Sales"]
}'
{"cube":"Sales","sql":"SELECT …","columns":[
   {"name":"Year","uniqueName":"[Time].[Year]","kind":"level"},
   {"name":"Unit Sales","uniqueName":"[Measures].[Unit Sales]","kind":"measure"},
   {"name":"Store Sales","uniqueName":"[Measures].[Store Sales]","kind":"measure"}],
 "rows":[[{"value":1997,"formatted":"1997"},
          {"value":266773,"formatted":"266773"},
          {"value":565238.13,"formatted":"565238.13"}]]}

Status codes: 400 invalid name (measure/dimension), 404 unknown cube, 503 no database.

POST /saiku/api/mdx/parse

Parses an MDX query and returns the AST (axes, slicer, formulas, canonical form).

$ curl -X POST localhost:8088/saiku/api/mdx/parse -H 'Content-Type: application/json' \
  -d '{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS, [Store].[Store Country].Members ON ROWS FROM [Sales]"}'
{"status":"PARSED","cube":"Sales","cubeKnown":true,
 "axes":[{"ordinal":0,"name":"COLUMNS","nonEmpty":false,"exp":"{[Measures].[Unit Sales]}"},
         {"ordinal":1,"name":"ROWS","nonEmpty":false,"exp":"[Store].[Store Country].Members"}],
 "slicer":"","canonical":"SELECT …"}

POST /saiku/api/mdx/execute

Parses, evaluates and returns the CellSet (with a grid for 2 axes).

$ curl -X POST localhost:8088/saiku/api/mdx/execute -H 'Content-Type: application/json' -d '{
  "mdx":"SELECT {[Measures].[Unit Sales],[Measures].[Store Sales]} ON COLUMNS, [Store].[Store Country].Members ON ROWS FROM [Sales]"

            "rows":[["266773","565238.13"]]}}

The engine covers a broad subset of MDX. The tables below list what is supported, with examples and the expected result over FoodMart (1997).

[WITH <formulas>]
SELECT <set> [NON EMPTY] ON COLUMNS [, <set> ON ROWS …]
FROM [Cube]
[WHERE <slicer>]

8.4 Calculated members (WITH MEMBER)

Arithmetic over measures and scalar functions, evaluated in Go per cell.

FeatureExampleResult (USA, 1997)
ArithmeticWITH MEMBER [Measures].[Profit] AS [Measures].[Store Sales] - [Measures].[Store Cost]339610.90
IIf(cond, x, y)IIf([Measures].[Unit Sales] > 80000, 1, 0)1 / 0
CoalesceEmpty(a, b)CoalesceEmpty([Measures].[X],[Measures].[Store Sales])first defined
Sum / Avg / Count / Aggregate (over a set)[Measures].[Unit Sales] / Sum([Store].[Store State].Members,[Measures].[Unit Sales]) * 100% of total (CA 28.02, OR 25.36, WA 46.62)
Subtotal in the right context. In Sum(set, measure), the subtotal is pre-computed grouped by the grid's other dimensions (not a global scalar). E.g. in a Gender × State grid, the % by state sums to 100% within each gender.

8.5 Operators and precedence

From lowest to highest precedence: OR / XOR / : → AND → NOT → comparisons (= <> < > <= >=, IS) → + - || → * / → unary + - → . (member/method). Supports CASE … WHEN … THEN … ELSE … END and CAST.

9. End-to-end examples

9.1 Breakdown by dimension (records)

POST /saiku/api/query
{"cube":"Sales","rows":[{"dimension":"Customers","level":"Country"}],
 "measures":["Unit Sales","Store Sales","Sales Count","Customer Count"]}

→ USA | 266773 | 565238.13 | 86837 | 5581

9.2 2-D grid via MDX

POST /saiku/api/mdx/execute
{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS,
         CrossJoin([Gender].[Gender].Members,[Marital Status].[Marital Status].Members) ON ROWS
         FROM [Sales]"}

→ F / M : 65336      F / S : 66222
  M / M : 66460      M / S : 68755     (sum = 266773)

9.3 Composable Top-N + ordering

{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS,
         Head(Order([Store].[Store State].Members,[Measures].[Unit Sales],BDESC),2) ON ROWS
         FROM [Sales]"}

→ WA : 124366      CA : 74748

9.4 Percent of total (calculated member + Sum over a set)

{"mdx":"WITH MEMBER [Measures].[Pct] AS
           [Measures].[Unit Sales] / Sum([Store].[Store State].Members,[Measures].[Unit Sales]) * 100
         SELECT {[Measures].[Unit Sales],[Measures].[Pct]} ON COLUMNS,
                [Store].[Store State].Members ON ROWS FROM [Sales]"}

→ CA : 74748 | 28.02      OR : 67659 | 25.36      WA : 124366 | 46.62   (Σ ≈ 100%)

9.5 Show all members vs NON EMPTY

… [Store].[Store State].Members ON ROWS …            → 10 rows (BC, DF, Jalisco… empty)
… NON EMPTY [Store].[Store State].Members ON ROWS …  → 3 rows (CA, OR, WA)

10. Schema formats

10.1 Mondrian XML (v3)

<Schema name="FoodMart">
  <Dimension name="Store">
    <Hierarchy hasAll="true" primaryKey="store_id">
      <Table name="store"/>
      <Level name="Store Country" column="store_country" uniqueMembers="true"/>
      <Level name="Store State"   column="store_state"/>
    </Hierarchy>
  </Dimension>
  <Cube name="Sales" defaultMeasure="Unit Sales">
    <Table name="sales_fact_1997"/>
    <DimensionUsage name="Store" source="Store" foreignKey="store_id"/>
    <Measure name="Unit Sales" column="unit_sales" aggregator="sum"/>
  </Cube>
</Schema>

10.2 Authoring YAML

schema: FoodMart
cubes:
  - name: Sales
    fact: sales_fact_1997
    defaultMeasure: Unit Sales
    measures:
      - {name: Unit Sales, column: unit_sales, agg: sum}
      - {name: Store Sales, column: store_sales, agg: sum, format: "#,###.00"}
    dimensions:
      - name: Time
        foreignKey: time_id
        table: time_by_day
        primaryKey: time_id
        type: time
        levels:
          - {name: Year, column: the_year, type: Numeric, levelType: TimeYears}
          - {name: Quarter, column: quarter, levelType: TimeQuarters}

Load either via CUBODW_SCHEMA=/path/file.xml (or .yml).

11. Limitations (not yet supported)

The engine is functional and covers most BI queries, but it does not have full Mondrian parity. Unsupported constructs return a clear error:

  • Min/Max over sets in calculated members.
  • Showing all members in CrossJoin (multi-binding still enumerates via the fact).
  • Named sets (WITH SET), ranges (m1 : m2), time functions (YTD, ParallelPeriod, Lag/Lead).
  • Snowflake dimensions (defined by <Join>, e.g. Product → product_class).
  • Multiple hierarchies per dimension, parent-child members.
  • Dialects other than PostgreSQL.
