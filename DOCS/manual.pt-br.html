CuboDW — Manual técnico

Motor OLAP (estilo Mondrian) escrito em Go: interpreta MDX, mapeia o modelo multidimensional para um star schema relacional, gera SQL (PostgreSQL) e devolve resultados tipados. Implantação em microserviços Docker.

Como ler este manual. A seção 2 dá o mapa das camadas. As seções 5 e 6 documentam cada módulo (pacote Go) com seus tipos, funções e exemplos. A 7 descreve a API REST e a 8 a cobertura de MDX. Os números dos exemplos são reais, calculados sobre o cubo de demonstração FoodMart (ano de 1997).

1. Visão geral

CuboDW transforma perguntas analíticas (em MDX ou num modelo de consulta JSON tipado) em SQL executado num data warehouse PostgreSQL, devolvendo uma matriz de células tipadas (um CellSet). O cubo é descrito por um schema (Mondrian XML ou YAML) que liga conceitos lógicos (dimensões, hierarquias, níveis, medidas) a tabelas e colunas físicas.

ItemValor
LinguagemGo 1.26
Banco-alvoPostgreSQL 16 (dialeto de referência)
ImplantaçãoDocker / docker compose (engine + postgres)
Linguagem de consultaMDX (subconjunto amplo) + modelo de query JSON
SchemaMondrian XML v3 e/ou YAML de autoria
LicençaApache-2.0
Módulo Gocubodw

2. Arquitetura

O código é um único módulo Go em três camadas: engine (o "Mondrian"), service (a lógica estilo Saiku) e web (HTTP/REST). O fluxo de uma consulta MDX percorre as camadas de cima para baixo:

   MDX (texto)
       │  engine/mdx  ............... lexer + parser  →  AST
       ▼
   service/mdxeval  .............. resolve eixos/membros, reduz ao modelo de query
       │  engine/query  ............ {medidas, níveis em linhas/colunas, filtros}
       ▼
   engine/sql  ................... gera SQL (star schema, JOIN fato↔dimensão)
       │  service/queryexec  ....... executa no PostgreSQL (pgx)
       ▼
   PostgreSQL  →  records  →  service/mdxeval (pivot)  →  CellSet
       ▲
   engine/metadata  ............. IR do cubo (alimentada por schema/mondrian|yaml)
CamadaPacotesResponsabilidade
enginemetadata, schema/mondrian, schema/yaml, query, sql, mdxModelo multidimensional, parsing MDX, geração de SQL
servicediscover, queryexec, mdxevalDescoberta de metadados, execução, avaliação MDX → CellSet
webwebServidor HTTP, endpoints /saiku/api/*
apoioconfig, version, demoConfiguração via env, versão, schema FoodMart embutido

3. Instalação e execução

Só é necessário Docker + Docker Compose no host. Go e Java não são necessários: o build do Go roda em container.

Subir a stack

make up            # docker compose: engine + postgres:16 (FoodMart auto-carregado)
curl localhost:8088/health        # {"status":"ok"}
curl localhost:8088/ready         # {"status":"ready"}  (conectado ao Postgres)
curl localhost:8088/saiku/api/info
make down          # derruba a stack

CLI (cmd/cubodw)

ComandoDescrição
cubodw serve-engineSobe o servidor HTTP do motor (flag --addr)
cubodw healthcheck --url …GET no endpoint de saúde; sai 0 se 200 (usado pelo HEALTHCHECK do Docker)
cubodw versionImprime a versão

Configuração (variáveis de ambiente — pacote config)

VariávelDefaultDescrição
CUBODW_HTTP_ADDR:8080Endereço de escuta HTTP (no container)
CUBODW_PG_DSN (ou PG_DSN)—DSN do PostgreSQL. Vazio: /ready reporta no-db
CUBODW_SCHEMA—Caminho do schema (.xml=Mondrian, .yml/.yaml=YAML). Vazio: FoodMart embutido
CUBODW_CONNECTIONfoodmartNome lógico da conexão exposto na descoberta
O host expõe o engine em localhost:8088 (mapeado para a porta 8080 do container). O PostgreSQL fica em localhost:5432 (usuário/senha cubodw, banco foodmart).

4. Conceitos OLAP (glossário)

TermoSignificado
CuboConjunto de medidas analisáveis por dimensões. Mapeia para uma tabela fato.
DimensãoEixo de análise (Tempo, Produto, Loja). Mapeia para uma tabela de dimensão ligada à fato por uma chave estrangeira.
HierarquiaSequência de níveis dentro de uma dimensão (ex.: País → Estado → Cidade).
NívelUm grau da hierarquia (ex.: "Store State"). Mapeia para uma coluna.
MembroUm valor de um nível (ex.: "CA", "WA").
MedidaValor numérico agregável (ex.: Unit Sales = soma de unidades). Vive na dimensão especial [Measures].
MDXMultiDimensional eXpressions: linguagem de consulta de cubos.
Star schemaModelagem física: uma tabela fato central + tabelas de dimensão ligadas por FK=PK.
CellSetResultado de uma consulta MDX: eixos (com posições) e células (valores).
Nome único (uniqueName)Identificador MDX entre colchetes, ex.: [Time].[Year], [Measures].[Unit Sales].

5. Módulos do motor

5.1 engine metadata

internal/engine/metadata

Define a IR (representação interna) do modelo multidimensional, alvo comum dos loaders e fonte da descoberta e da geração de SQL. Hierarquia de tipos: Schema → Cube → Dimension → Hierarchy → Level (+ Property) e, em paralelo, Cube → Measure.

Tipos principais

TipoCampos relevantes
SchemaName, Cubes []*Cube, SharedDimensions []*Dimension
CubeName, Caption, DefaultMeasure, Fact Relation, Dimensions, Measures
DimensionName, Type (StandardDimension|TimeDimension), ForeignKey, Hierarchies
HierarchyName, HasAll, AllMemberName, PrimaryKey, Table Relation, Levels
LevelName, Column, Type, LevelType, UniqueMembers, Properties
MeasureName, Column, Aggregator (sum|count|min|max|avg|distinct-count), FormatString, Expression
RelationSchema, Table, Alias — referência a uma tabela física

Funções

5.2 engine schema/mondrian

internal/engine/schema/mondrian

Carrega schemas Mondrian (XML v3) para a IR. Resolve DimensionUsage contra dimensões compartilhadas, dimensões inline, e MeasureExpression (prefere o dialeto postgres).

Funções

func LoadFile(path string) (*metadata.Schema, error)
func LoadBytes(b []byte)   (*metadata.Schema, error)
func Load(r io.Reader)     (*metadata.Schema, error)

Exemplo

schema, err := mondrian.LoadFile("FoodMart.xml")
cube, _ := schema.FindCube("Sales")
// cube.Fact.Table == "sales_fact_1997"
// cube.Measures: Unit Sales, Store Cost, Store Sales, Sales Count, Customer Count, ...
// cube.Dimensions: Store, Time, Product, Customers, Gender, ...

5.4 engine query

internal/engine/query

Define a especificação de uma consulta tabular (pivot) e o formato do resultado tipado. É o precursor do "thin query": descreve medidas, níveis em linhas/colunas e filtros sem expor MDX. O avaliador MDX (seção 6.3) produz esta mesma estrutura.

type Query struct {
    Cube     string       `json:"cube"`
    Measures []string     `json:"measures"`
    Rows     []LevelRef   `json:"rows,omitempty"`
    Columns  []LevelRef   `json:"columns,omitempty"`
    Filters  []Filter     `json:"filters,omitempty"`
}
type LevelRef struct { Dimension, Hierarchy, Level string }
type Filter   struct { Dimension, Hierarchy, Level string; Members []string }

type Result struct {
    Cube    string     `json:"cube"`
    SQL     string     `json:"sql"`      // SQL gerada (transparência)
    Columns []Column   `json:"columns"`  // {name, uniqueName, kind: level|measure}
    Rows    [][]Cell   `json:"rows"`     // {value, formatted}
}

5.5 engine sql

internal/engine/sql

Gera SQL relacional a partir de uma query.Query sobre um cubo (star schema). Dialeto atual: PostgreSQL.

Tipos e funções

type Dialect interface { QuoteIdent(name string) string; Name() string }
type Postgres struct{}                       // dialeto PostgreSQL
type Statement struct { SQL string; Args []any; Columns []query.Column }

func Build(d Dialect, cube *metadata.Cube, q query.Query) (*Statement, error)
func BuildLevelMembers(d Dialect, cube *metadata.Cube, ref query.LevelRef, filters []query.Filter) (*Statement, error)

Exemplo — Build

Query: medida Unit Sales por nível Time.Year →

SELECT "Time"."the_year", sum("f"."unit_sales")::float8
FROM "sales_fact_1997" AS "f"
JOIN "time_by_day" AS "Time" ON "f"."time_id" = "Time"."time_id"
GROUP BY "Time"."the_year"
ORDER BY 1

Filtros viram predicados (coluna)::text = ANY($n); medidas numéricas saem como float8 e contagens como inteiro. Dimensões em snowflake (via <Join>) retornam erro claro.

Exemplo — BuildLevelMembers

Enumera todos os membros de um nível a partir da tabela de dimensão (base da semântica "mostrar todos os membros"):

SELECT DISTINCT "Store"."store_state"
FROM "store" AS "Store"
WHERE ("Store"."store_country")::text = ANY($1)   -- p.ex. ['USA']
ORDER BY 1

5.6 engine mdx

internal/engine/mdx

Implementa o lexer, AST e parser da linguagem MDX, fiel à gramática do Mondrian (MdxParser.jj). Faz apenas o parsing (texto → AST); a avaliação está em mdxeval.

Funções de entrada

func Tokenize(input string)        ([]Token, error)   // lexer
func ParseQuery(input string)      (*Query, error)    // SELECT … FROM … WHERE …
func ParseExpression(input string) (Exp, error)       // expressão isolada

AST

Tudo é uma Exp. Chamadas de função/operadores viram FunCall{Name, Syntax, Args}; identificadores são Id com segmentos. Decisão de design fiel ao Mondrian: membros como [Measures].[Unit Sales] e [Store].[USA].Children são um único Id multi-segmento; só viram FunCall quando há parênteses (método/função).

TipoDescrição
QueryFormulas []*Formula (WITH), Axes []*Axis, Cube string, Slicer Exp (WHERE)
AxisNonEmpty bool, Exp, Ordinal int (0=COLUMNS,1=ROWS,…)
FormulaIsMember bool, Name *Id, Exp
FunCallName, Syntax (Function/Method/Property/Infix/Prefix/Postfix/Braces/Parentheses/Case/Cast), Args []Exp
IdSegments []Segment (Name, Quoted, Key)
LiteraisNumericLiteral, StringLiteral, NullLiteral

Cada nó tem String() que reconstrói uma forma canônica (útil para depurar e para gerar MDX).

Exemplo

q, _ := mdx.ParseQuery(`SELECT {[Measures].[Unit Sales]} ON COLUMNS,
                        [Store].[Store Country].Members ON ROWS
                        FROM [Sales] WHERE [Time].[1997]`)
// q.Cube == "Sales";  len(q.Axes) == 2
// q.Axes[1].Exp.String() == "[Store].[Store Country].Members"
// q.Slicer.String()      == "[Time].[1997]"

6.2 service queryexec

internal/service/queryexec

Planeja (gera SQL) e executa queries de cubo contra o PostgreSQL, devolvendo um query.Result tipado.

func New(pool *pgxpool.Pool) *Service
func (s *Service) Plan(cube, q query.Query)              (*sql.Statement, error)  // só gera SQL (validação)
func (s *Service) Run(ctx, cube, st *sql.Statement)      (*query.Result, error)   // executa
func (s *Service) EnumerateLevel(ctx, cube, ref, filters) ([]string, error)       // membros via dimensão
func (s *Service) HasDB() bool                                                     // há conexão?

Medidas numéricas (somas) voltam como float8 → número JSON; contagens como inteiro. EnumerateLevel lista os membros de um nível a partir da tabela de dimensão (não do fato).

6.3 service mdxeval

internal/service/mdxeval

O avaliador MDX. Reduz o AST ao modelo de query (níveis + medidas + filtros), executa via queryexec e pivota os records num CellSet. Estratégia: resolve as posições de cada eixo independentemente (no contexto do slicer), o que permite ordenação/limite/filtro por medida fiéis ao Mondrian e cálculo de membros calculados.

func Evaluate(ctx, cube *metadata.Cube, q *mdx.Query, exec *queryexec.Service) (*CellSet, error)

Resultado — CellSet

type CellSet struct {
    Cube  string     `json:"cube"`
    SQL   string     `json:"sql"`
    Axes  []Axis     `json:"axes"`    // {ordinal, name, positions:[{members:[{caption,uniqueName}]}]}
    Cells []Cell     `json:"cells"`   // {coords:[…], value, formatted}
    Grid  *Grid      `json:"grid"`    // conveniência p/ 2 eixos: columnHeaders, rowHeaders, rows[][]
}

7. API REST

Todas as rotas ficam sob /saiku/api/* (contrato compatível com o Saiku, para plugar uma UI no futuro). Base local: http://localhost:8088.

GET /health · /ready · /saiku/api/info

Saúde (liveness), prontidão (pinga o Postgres) e metadados do serviço.

$ curl localhost:8088/saiku/api/info
{"name":"cubodw-engine","version":"0.0.1","engine":"mondrian-go (em construção)",
 "connection":"foodmart","schema":"FoodMart","cubes":7}

GET /saiku/api/discover

Árvore conexão → catálogo → schema → cubos.

$ curl localhost:8088/saiku/api/discover
[{"name":"foodmart","catalogs":[{"name":"FoodMart","schemas":[{"name":"FoodMart",
  "cubes":[{"name":"Sales","uniqueName":"[Sales]","caption":"Sales", … }, … ]}]}]}]

GET /saiku/api/discover/{conn}/{catalog}/{schema}/{cube}/metadata

Dimensões (com hierarquias e níveis) e medidas de um cubo.

$ curl localhost:8088/saiku/api/discover/foodmart/FoodMart/FoodMart/Sales/metadata
{"name":"Sales","uniqueName":"[Sales]","defaultMeasure":"Unit Sales",
 "dimensions":[{"name":"Time","hierarchies":[{"levels":[
     {"name":"Year","uniqueName":"[Time].[Year]"}, … ]}]}, … ],
 "measures":[{"name":"Unit Sales","uniqueName":"[Measures].[Unit Sales]", … }, … ]}

Há também …/{cube}/dimensions (só a lista de dimensões).

POST /saiku/api/query · /saiku/api/query/preview

Executa (ou só valida + gera SQL, em /preview) o modelo de query JSON (sem MDX).

$ curl -X POST localhost:8088/saiku/api/query -H 'Content-Type: application/json' -d '{
  "cube":"Sales",
  "rows":[{"dimension":"Time","level":"Year"}],
  "measures":["Unit Sales","Store Sales"]

O motor cobre um subconjunto amplo de MDX. As tabelas abaixo listam o que é suportado, com exemplos e o resultado esperado sobre o FoodMart (1997).

8.4 Membros calculados (WITH MEMBER)

Aritmética sobre medidas e funções escalares, avaliadas em Go por célula.

RecursoExemploResultado (USA, 1997)
AritméticaWITH MEMBER [Measures].[Profit] AS [Measures].[Store Sales] - [Measures].[Store Cost]339610.90
IIf(cond, x, y)IIf([Measures].[Unit Sales] > 80000, 1, 0)1 / 0
CoalesceEmpty(a, b)CoalesceEmpty([Measures].[X],[Measures].[Store Sales])primeiro definido
Sum / Avg / Count / Aggregate (sobre conjunto)[Measures].[Unit Sales] / Sum([Store].[Store State].Members,[Measures].[Unit Sales]) * 100% do total (CA 28.02, OR 25.36, WA 46.62)
Subtotal no contexto correto. Em Sum(set, medida), o subtotal é pré-computado agrupado pelas demais dimensões do grid (não um escalar global). Ex.: num grid Gênero × Estado, o % por estado soma 100% dentro de cada gênero.

8.5 Operadores e precedência

Da menor para a maior precedência: OR / XOR / : → AND → NOT → comparações (= <> < > <= >=, IS) → + - || → * / → unário + - → . (membro/método). Suporta CASE … WHEN … THEN … ELSE … END e CAST.

9. Exemplos ponta a ponta

9.1 Detalhamento por dimensão (records)

POST /saiku/api/query
{"cube":"Sales","rows":[{"dimension":"Customers","level":"Country"}],
 "measures":["Unit Sales","Store Sales","Sales Count","Customer Count"]}

→ USA | 266773 | 565238.13 | 86837 | 5581

9.2 Grade 2-D via MDX

POST /saiku/api/mdx/execute
{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS,
         CrossJoin([Gender].[Gender].Members,[Marital Status].[Marital Status].Members) ON ROWS
         FROM [Sales]"}

→ F / M : 65336      F / S : 66222
  M / M : 66460      M / S : 68755     (soma = 266773)

9.3 Top-N + ordenação compostas

{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS,
         Head(Order([Store].[Store State].Members,[Measures].[Unit Sales],BDESC),2) ON ROWS
         FROM [Sales]"}

→ WA : 124366      CA : 74748

9.4 Percentual do total (membro calculado + Sum de conjunto)

{"mdx":"WITH MEMBER [Measures].[Pct] AS
           [Measures].[Unit Sales] / Sum([Store].[Store State].Members,[Measures].[Unit Sales]) * 100
         SELECT {[Measures].[Unit Sales],[Measures].[Pct]} ON COLUMNS,
                [Store].[Store State].Members ON ROWS FROM [Sales]"}

→ CA : 74748 | 28.02      OR : 67659 | 25.36      WA : 124366 | 46.62   (Σ ≈ 100%)

9.5 Mostrar todos os membros vs NON EMPTY

… [Store].[Store State].Members ON ROWS …            → 10 linhas (BC, DF, Jalisco… vazias)
… NON EMPTY [Store].[Store State].Members ON ROWS …  → 3 linhas (CA, OR, WA)

10. Formatos de schema

10.1 Mondrian XML (v3)

<Schema name="FoodMart">
  <Dimension name="Store">
    <Hierarchy hasAll="true" primaryKey="store_id">
      <Table name="store"/>
      <Level name="Store Country" column="store_country" uniqueMembers="true"/>
      <Level name="Store State"   column="store_state"/>
    </Hierarchy>
  </Dimension>
  <Cube name="Sales" defaultMeasure="Unit Sales">
    <Table name="sales_fact_1997"/>
    <DimensionUsage name="Store" source="Store" foreignKey="store_id"/>
    <Measure name="Unit Sales" column="unit_sales" aggregator="sum"/>
  </Cube>
</Schema>

10.2 YAML de autoria

schema: FoodMart
cubes:
  - name: Sales
    fact: sales_fact_1997
    defaultMeasure: Unit Sales
    measures:
      - {name: Unit Sales, column: unit_sales, agg: sum}
      - {name: Store Sales, column: store_sales, agg: sum, format: "#,###.00"}
    dimensions:
      - name: Time
        foreignKey: time_id
        table: time_by_day
        primaryKey: time_id
        type: time
        levels:
          - {name: Year, column: the_year, type: Numeric, levelType: TimeYears}
          - {name: Quarter, column: quarter, levelType: TimeQuarters}

Carregue qualquer um via CUBODW_SCHEMA=/caminho/arquivo.xml (ou .yml).

11. Limitações (ainda não suportado)

O motor é funcional e cobre a maioria das consultas de BI, mas não tem paridade total com o Mondrian. Construções não suportadas retornam erro claro:

  • Min/Max sobre conjuntos em membros calculados.
  • Mostrar todos os membros em CrossJoin (multi-binding ainda enumera via fato).
  • Named sets (WITH SET), ranges (m1 : m2), funções de tempo (YTD, ParallelPeriod, Lag/Lead).
  • Dimensões em snowflake (definidas por <Join>, ex.: Product → product_class).
  • Múltiplas hierarquias por dimensão, membros pai-filho.
  • Dialetos além do PostgreSQL.
