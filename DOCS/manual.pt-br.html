CuboDW — Manual técnico

Motor OLAP (estilo Mondrian) escrito em Go: interpreta MDX, mapeia o modelo multidimensional para um star schema relacional, gera SQL (5 dialetos), devolve resultados tipados e inclui uma aplicação visual drag-and-drop com autenticação. Implantação em microserviços Docker.

Como ler este manual. A seção 2 dá o mapa das camadas. As seções 5 e 6 documentam cada módulo (pacote Go). A 7 cobre autenticação, a 8 a API REST, a 9 a cobertura MDX, a 10 a interface visual e a 11 os dialetos SQL. Os números dos exemplos são reais, calculados sobre o cubo de demonstração FoodMart (ano de 1997).

1. Visão geral

CuboDW transforma perguntas analíticas (em MDX ou num modelo de consulta JSON tipado) em SQL executado num data warehouse PostgreSQL, devolvendo uma matriz de células tipadas (um CellSet). O cubo é descrito por um schema (Mondrian XML ou YAML) que liga conceitos lógicos (dimensões, hierarquias, níveis, medidas) a tabelas e colunas físicas. Por cima do motor há uma UI visual e uma camada de autenticação com papéis.

2. Arquitetura

O código é um único módulo Go em três camadas: engine (o "Mondrian"), service (a lógica estilo Saiku) e web (HTTP/REST + UI). O fluxo de uma consulta MDX percorre as camadas de cima para baixo:

   HTTP  →  web (middleware de autenticação)  →  handlers /saiku/api/*
       │
   MDX (texto)
       │  engine/mdx  ............... lexer + parser  →  AST
       ▼
   service/mdxeval  .............. resolve eixos/membros, reduz ao modelo de query
       │  engine/query  ............ {medidas, níveis em linhas/colunas, filtros}
       ▼
   engine/sql  ................... gera SQL (star schema, JOIN fato↔dimensão, por Dialect)
       │  service/queryexec  ....... executa no PostgreSQL (pgx) + cache de resultados
       ▼
   PostgreSQL  →  records  →  service/mdxeval (pivot)  →  CellSet
       ▲
   engine/metadata  ............. IR do cubo (alimentada por schema/mondrian|yaml)
CamadaPacotesResponsabilidade
enginemetadata, schema/mondrian, schema/yaml, query, sql (+ dialect), mdxModelo multidimensional, parsing MDX, geração de SQL multi-dialeto
servicediscover, queryexec, mdxevalDescoberta de metadados, execução + cache, avaliação MDX → CellSet
webweb (+ web/ui embutida)Servidor HTTP, autenticação (middleware), endpoints /saiku/api/*, SPA /ui/
apoioconfig, version, demo, authConfiguração via env, versão, FoodMart embutido, store de usuários + tokens

3. Instalação e execução

Só é necessário Docker + Docker Compose no host. Go e Java não são necessários: o build do Go roda em container.

Subir a stack

git clone https://github.com/davidkestering/projeto_golap.git && cd projeto_golap
make up            # docker compose: engine + postgres:16 (FoodMart auto-carregado)
curl localhost:8088/health        # {"status":"ok"}
curl localhost:8088/ready         # {"status":"ready"}  (conectado ao Postgres)
# 👉 UI drag-and-drop:  http://localhost:8088/ui/   (login demo: admin / admin)
make down          # derruba a stack

4. Conceitos OLAP (glossário)

TermoSignificado
CuboConjunto de medidas analisáveis por dimensões. Mapeia para uma tabela fato.
DimensãoEixo de análise (Tempo, Produto, Loja). Mapeia para uma tabela de dimensão ligada à fato por uma chave estrangeira.
HierarquiaSequência de níveis dentro de uma dimensão (ex.: País → Estado → Cidade). Uma dimensão pode ter várias hierarquias (ex.: Time padrão e Weekly).
NívelUm grau da hierarquia (ex.: "Store State"). Mapeia para uma coluna.
MembroUm valor de um nível (ex.: "CA", "WA").
MedidaValor numérico agregável (ex.: Unit Sales = soma de unidades). Vive na dimensão especial [Measures].
Membro calculadoMedida/membro definido por expressão em WITH MEMBER (ex.: Profit = Store Sales − Store Cost).
Named setConjunto nomeado definido em WITH SET e reutilizado nos eixos.
MDXMultiDimensional eXpressions: linguagem de consulta de cubos.
Star schemaModelagem física: uma tabela fato central + tabelas de dimensão ligadas por FK=PK.
CellSetResultado de uma consulta MDX: eixos (com posições) e células (valores).
Drill-throughDescer de uma célula agregada para as linhas de fato cruas que a compõem.
Nome único (uniqueName)Identificador MDX entre colchetes, ex.: [Time].[Year], [Measures].[Unit Sales].

5. Módulos do motor

5.1 engine metadata

internal/engine/metadata

Define a IR (representação interna) do modelo multidimensional, alvo comum dos loaders e fonte da descoberta e da geração de SQL. Hierarquia de tipos: Schema → Cube → Dimension → Hierarchy → Level (+ Property) e, em paralelo, Cube → Measure.

Tipos principais

TipoCampos relevantes
SchemaName, Cubes []*Cube, SharedDimensions []*Dimension
CubeName, Caption, DefaultMeasure, Fact Relation, Dimensions, Measures
DimensionName, Type (StandardDimension|TimeDimension), ForeignKey, Hierarchies
HierarchyName, HasAll, AllMemberName, PrimaryKey, Table Relation, Levels; EffectiveName(d) resolve hierarquia default vs nomeada
LevelName, Column, NameColumn, ParentColumn/NullParentValue (parent-child), Type, LevelType, UniqueMembers, Properties; DisplayColumn(), IsParentChild()
MeasureName, Column, Aggregator (sum|count|min|max|avg|distinct-count), FormatString, Expression
RelationSchema, Table, Alias — referência a uma tabela física

5.2 engine schema/mondrian

internal/engine/schema/mondrian

Carrega schemas Mondrian (XML v3) para a IR. Resolve DimensionUsage contra dimensões compartilhadas, dimensões inline, múltiplas hierarquias por dimensão, nameColumn/parentColumn (parent-child) e MeasureExpression (prefere o dialeto postgres).

func LoadFile(path string) (*metadata.Schema, error)
func LoadBytes(b []byte)   (*metadata.Schema, error)
func Load(r io.Reader)     (*metadata.Schema, error)

5.3 engine schema/yaml

internal/engine/schema/yaml

Carrega cubos no formato de autoria YAML (enxuto) para a mesma IR. Cada dimensão YAML é achatada numa hierarquia única (com membro "All"). Veja o formato completo na seção 13.

5.4 engine query

internal/engine/query

Define a especificação de uma consulta tabular (pivot) e o formato do resultado tipado. O avaliador MDX (seção 6.3) produz esta mesma estrutura. Inclui o formatador de medidas query.Format (formatString).

type Query struct {
    Cube     string       `json:"cube"`
    Measures []string     `json:"measures"`
    Rows     []LevelRef   `json:"rows,omitempty"`
    Columns  []LevelRef   `json:"columns,omitempty"`
    Filters  []Filter     `json:"filters,omitempty"`
    Totals   bool         `json:"totals,omitempty"`
}
type LevelRef struct { Dimension, Hierarchy, Level string }
type Filter   struct { Dimension, Hierarchy, Level string; Members []string }

type Column struct { Name, UniqueName, Kind, FormatString string }  // kind: level|measure
type Result struct { Cube, SQL string; Columns []Column; Rows [][]Cell; Totals []Cell }

// Format aplica um formatString estilo Mondrian/VBA: "#,###.00", "#,###",
// "Standard", "Currency", "0.0%", … (agrupamento de milhares, casas decimais).
func Format(v float64, formatString string) string

5.5 engine sql + dialetos

internal/engine/sql

Gera SQL relacional a partir de uma query.Query sobre um cubo (star schema). A geração é dirigida por um Dialect, que abstrai as diferenças entre bancos. Veja a lista de dialetos na seção 11.

type Dialect interface {
    Name() string
    QuoteIdent(name string) string                 // "x" | `x` | [x]
    Placeholder(i int) string                      // $1 | ? | @p1 | :1
    CastText(expr) string;  CastFloat(expr) string // ::text / CAST(... AS ...)
    InClause(col, members, startArg) (sql, args)   // ANY($n) ou IN (…)
    SelectTop(n) string;    LimitSuffix(n) string  // TOP n / LIMIT n / FETCH FIRST
    AliasSep() string                              // " AS " | " " (Oracle)
}
func Build(d Dialect, cube *metadata.Cube, q query.Query) (*Statement, error)
func BuildLevelMembers(d Dialect, cube, ref, filters) (*Statement, error)   // membros via dimensão
func BuildDrillthrough(d Dialect, cube, filters, maxrows) (*Statement, error) // linhas de fato cruas
func DialectByName(name string) (Dialect, error)

Exemplo — Build (PostgreSQL)

SELECT "Time"."the_year", sum("f"."unit_sales")::float8
FROM "sales_fact_1997" AS "f"
JOIN "time_by_day" AS "Time" ON "f"."time_id" = "Time"."time_id"
GROUP BY "Time"."the_year"
ORDER BY 1

Níveis com nameColumn (ex.: parent-child) exibem o nome em vez da chave via DisplayColumn(). Dimensões em snowflake (via <Join>) retornam erro claro.

5.6 engine mdx

internal/engine/mdx

Implementa o lexer, AST e parser da linguagem MDX, fiel à gramática do Mondrian (MdxParser.jj). Faz apenas o parsing (texto → AST); a avaliação está em mdxeval.

func Tokenize(input string)        ([]Token, error)   // lexer
func ParseQuery(input string)      (*Query, error)    // SELECT … FROM … WHERE …
func ParseExpression(input string) (Exp, error)       // expressão isolada

Decisão de design fiel ao Mondrian: membros como [Measures].[Unit Sales] e [Store].[USA].Children são um único Id multi-segmento; só viram FunCall quando há parênteses (método/função). Suporta WITH MEMBER/WITH SET, ranges (m1 : m2), CASE e CAST.

A formatação de cada célula de medida aplica o FormatString da coluna (ex.: 565,238.13).

6.3 service mdxeval

internal/service/mdxeval

O avaliador MDX. Reduz o AST ao modelo de query (níveis + medidas + filtros), executa via queryexec e pivota os records num CellSet. Resolve as posições de cada eixo independentemente (no contexto do slicer), o que permite ordenação/limite/filtro por medida, membros calculados, named sets, ranges, inteligência de tempo (time.go), múltiplas hierarquias e parent-child.

func Evaluate(ctx, cube *metadata.Cube, q *mdx.Query, exec *queryexec.Service) (*CellSet, error)
type CellSet struct {
    Cube  string; SQL string
    Axes  []Axis     // {ordinal, name, positions:[{members:[{caption,uniqueName}]}]}
    Cells []Cell     // {coords:[…], value, formatted}
    Grid  *Grid      // conveniência p/ 2 eixos: columnHeaders, rowHeaders, rows[][]
}

7. Autenticação e papéis

A autenticação está ligada por padrão (CUBODW_AUTH_ENABLED=true). Um middleware protege todos os endpoints sob /saiku/api/*, exceto os públicos. A sessão é um cookie assinado (HMAC) — sem estado no servidor; as senhas usam bcrypt.

RecursoDetalhe
Papéisadmin (tudo) e user (consulta/descoberta)
Admin semeadoadmin / admin (troque em produção)
Registroaberto: qualquer um cria um usuário user e já é autenticado
Rotas públicas/health, /ready, /saiku/api/info, /saiku/api/auth/*, /ui/*
Admin-onlyAções administrativas (ex.: POST /saiku/api/cache/clear) → 403 para user
Cookiecubodw_session, HttpOnly, SameSite=Lax, TTL 12h

Fluxo (cURL com cookies)

# sem login, endpoint protegido → 401
curl -i localhost:8088/saiku/api/discover                       # HTTP 401

# login (guarda o cookie de sessão)
curl -c cookies.txt -X POST localhost:8088/saiku/api/auth/login \
  -H 'Content-Type: application/json' -d '{"username":"admin","password":"admin"}'
# {"username":"admin","role":"admin"}

# com o cookie, a API responde
curl -b cookies.txt localhost:8088/saiku/api/discover           # HTTP 200

# registro aberto (cria papel user, já logado)
curl -X POST localhost:8088/saiku/api/auth/register \
  -H 'Content-Type: application/json' -d '{"username":"maria","password":"s3nha"}'
# {"username":"maria","role":"user"}
Produção. Defina CUBODW_AUTH_SECRET (senão um segredo de desenvolvimento é usado, com aviso no log), persista usuários com CUBODW_USERS_FILE, troque a senha do admin e sirva atrás de HTTPS (para habilitar o atributo Secure no cookie).

8. API REST

Todas as rotas ficam sob /saiku/api/* (contrato compatível com o Saiku). Base local: http://localhost:8088. Com a auth ligada, envie o cookie de sessão (seção 7) — os exemplos abaixo omitem o -b cookies.txt por brevidade.

GET /health · /ready · /saiku/api/info

Saúde (liveness), prontidão (pinga o Postgres) e metadados do serviço (inclui cache). Públicos.

POST /saiku/api/auth/register · /login · /logout · GET /me

Registro aberto (cria user), login (cookie), logout (limpa o cookie) e identidade atual.

GET /saiku/api/discover · …/{cube}/metadata

Árvore conexão → catálogo → schema → cubos e, por cubo, dimensões (com hierarquias e níveis) e medidas.

POST /saiku/api/query · /saiku/api/query/preview

Executa (ou só valida + gera SQL, em /preview) o modelo de query JSON (sem MDX). Com "totals":true devolve também a linha de totais.

$ curl -X POST localhost:8088/saiku/api/query -H 'Content-Type: application/json' -d '{
  "cube":"Sales", "rows":[{"dimension":"Time","level":"Year"}],
  "measures":["Unit Sales","Store Sales"]
}'
{"cube":"Sales","sql":"SELECT …","columns":[ … ],
 "rows":[[{"value":1997,"formatted":"1997"},
          {"value":266773,"formatted":"266,773"},
          {"value":565238.13,"formatted":"565,238.13"}]]}

POST /saiku/api/query/drillthrough

Devolve as linhas de fato cruas por trás de uma célula agregada (filtros = membros da célula; maxrows limita).

{"cube":"Sales","filters":[{"dimension":"Store","level":"Store State","members":["CA"]}],"maxrows":50}

POST /saiku/api/mdx/parse · /saiku/api/mdx/execute

parse devolve o AST (eixos, slicer, fórmulas, forma canônica). execute faz o parse, avalia e devolve o CellSet (com grid para 2 eixos).

GET /saiku/api/ai/cubes · /ai/schema/{cube} · POST /ai/query

AI Query API: consulta cubos sem MDX, com schema auto-descritivo (membros de amostra) e validação tipada. Em erro de nome, devolve {status, field, available:[…]} para autocorreção.

$ curl -X POST localhost:8088/saiku/api/ai/query -H 'Content-Type: application/json' -d '{
  "cube":"Sales","measures":["Unit Sales"],"rows":["Store.Store State"],
  "filters":{"Time.Year":["1997"]}}'

GET /saiku/api/schemas · POST /schemas/validate · /schemas (admin) · DELETE /schemas/{name} (admin)

Gerenciador de cubos: lista, valida (dry-run), adiciona e remove schemas em tempo de execução. Corpo: {"content":"…","format":"yaml|xml"} (YAML de autoria ou Mondrian XML). Colisão de nome de cubo/schema → 409.

GET /saiku/api/cache · POST /saiku/api/cache/clear (admin)

Métricas do cache (hits, misses, size) e limpeza (papel admin).

Códigos: 400 nome inválido · 401 não autenticado · 403 requer admin · 404 cubo inexistente · 503 sem banco.

O motor cobre um subconjunto amplo de MDX. As tabelas listam o que é suportado, com exemplos e resultado esperado sobre o FoodMart (1997).

9.7 Formatação de medidas

O formatString de cada medida é aplicado no resultado: Unit Sales (Standard) → 266,773; Store Sales (#,###.00) → 565,238.13; Sales Count (#,###) → 86,837.

9.8 Operadores e precedência

Da menor para a maior precedência: OR / XOR / : → AND → NOT → comparações → + - || → * / → unário + - → . (membro/método). Suporta CASE … WHEN … THEN … ELSE … END e CAST.

11. Dialetos SQL

A geração de SQL é dirigida por um Dialect (seção 5.5). Cinco dialetos estão implementados e cobertos por golden tests. O runtime executa via pgx (PostgreSQL); apontar para os demais requer plugar o driver respectivo e carregar os dados nesse banco (passo de deploy) — a camada de geração de SQL já está pronta.

DialetoQuotingPlaceholderTexto / FloatLimite
PostgreSQL"x"$1(e)::text / (e)::float8LIMIT n
MySQL/MariaDB`x`?CAST(e AS CHAR/DOUBLE)LIMIT n
DuckDB"x"?CAST(e AS VARCHAR/DOUBLE)LIMIT n
SQL Server[x]@p1CAST(e AS NVARCHAR/FLOAT)SELECT TOP n
Oracle"x":1CAST(e AS VARCHAR2/BINARY_DOUBLE)FETCH FIRST n ROWS ONLY

Resolva por nome com sql.DialectByName("mysql"|"duckdb"|"sqlserver"|"oracle"|"postgres"). O Oracle, por exemplo, omite o AS antes do alias de tabela (FROM "t" "f").

12. Exemplos ponta a ponta

12.1 Detalhamento por dimensão

POST /saiku/api/query
{"cube":"Sales","rows":[{"dimension":"Customers","level":"Country"}],
 "measures":["Unit Sales","Store Sales","Sales Count","Customer Count"]}
→ USA | 266,773 | 565,238.13 | 86,837 | 5,581

12.2 Grade 2-D via MDX (CrossJoin)

{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS,
         CrossJoin([Gender].[Gender].Members,[Marital Status].[Marital Status].Members) ON ROWS
         FROM [Sales]"}
→ F/M:65336  F/S:66222  M/M:66460  M/S:68755   (soma 266773)

12.3 Percentual do total (membro calculado + Sum)

{"mdx":"WITH MEMBER [Measures].[Pct] AS
           [Measures].[Unit Sales] / Sum([Store].[Store State].Members,[Measures].[Unit Sales]) * 100
         SELECT {[Measures].[Unit Sales],[Measures].[Pct]} ON COLUMNS,
                [Store].[Store State].Members ON ROWS FROM [Sales]"}
→ CA:74748|28.02   OR:67659|25.36   WA:124366|46.62   (Σ ≈ 100%)

12.4 Hierarquia nomeada (Weekly)

{"mdx":"SELECT {[Measures].[Unit Sales]} ON COLUMNS,
         [Time].[Weekly].[Week].Members ON ROWS
         FROM [Sales] WHERE [Time].[1997]"}
→ 52 semanas (1..52)   — vs [Time].[Quarter].Members → 4 trimestres

12.5 Mostrar todos os membros vs NON EMPTY

… [Store].[Store State].Members ON ROWS …            → 10 linhas (vazias incluídas)
… NON EMPTY [Store].[Store State].Members ON ROWS …  → 3 linhas (CA, OR, WA)

13. Formatos de schema

13.1 Mondrian XML (v3)

<Schema name="FoodMart">
  <Dimension name="Store">
    <Hierarchy hasAll="true" primaryKey="store_id">
      <Table name="store"/>
      <Level name="Store Country" column="store_country" uniqueMembers="true"/>
      <Level name="Store State"   column="store_state"/>
    </Hierarchy>
  </Dimension>
  <Cube name="Sales" defaultMeasure="Unit Sales">
    <Table name="sales_fact_1997"/>
    <DimensionUsage name="Store" source="Store" foreignKey="store_id"/>
    <Measure name="Unit Sales" column="unit_sales" aggregator="sum" formatString="Standard"/>
  </Cube>
</Schema>

13.2 YAML de autoria

schema: FoodMart
cubes:
  - name: Sales
    fact: sales_fact_1997
    defaultMeasure: Unit Sales
    measures:
      - {name: Unit Sales, column: unit_sales, agg: sum}
      - {name: Store Sales, column: store_sales, agg: sum, format: "#,###.00"}
    dimensions:
      - name: Time
        foreignKey: time_id
        table: time_by_day
        primaryKey: time_id
        type: time
        levels:
          - {name: Year, column: the_year, type: Numeric, levelType: TimeYears}
          - {name: Quarter, column: quarter, levelType: TimeQuarters}

Carregue um schema fixo no boot via CUBODW_SCHEMA=/caminho/arquivo.xml (ou .yml), ou adicione cubos em tempo de execução pela UI (botão ⬢ Cubos, admin) / API POST /saiku/api/schemas — ver seções 8 e 10. Para persistir os adicionados entre reinícios, defina CUBODW_SCHEMAS_DIR.

14. Limitações (ainda não suportado)

O motor é funcional e cobre a maioria das consultas de BI, mas não tem paridade total com o Mondrian. Construções não suportadas retornam erro claro:

  • Mostrar todos os membros em CrossJoin (multi-binding ainda enumera via fato).
  • Dimensões em snowflake (definidas por <Join>, ex.: Product → product_class).
  • Parent-child apenas como lista plana (por nome): navegação de árvore e rollup recursivo ainda não.
  • Execução em runtime fora do PostgreSQL: os dialetos MySQL/DuckDB/SQL Server/Oracle existem na geração de SQL, mas o runtime usa pgx (plugar outro driver + dados é um passo de deploy).
